## Current state: MVP Step 1 — notebook document + save pipeline

Built on the Phase 0 bridge spike. Full product/tech spec lives outside the repo at `firstmate/data/cellar-spec.md` (read §3 for the versioning model); nbdev field-policy detail at `firstmate/data/nbdev-study-n8/report.md`. See `README.md` for run/verify details.

- **Two runtimes.** Node (SvelteKit) + a Python sidecar (Jupyter kernel service). Both are booted by the CLI launcher `bin/cellar.js` (one command → both servers up → browser opens, workspace = cwd). Distributed as an **npm package** (`package.json` `files`=`build/`+`bin/`+`src/lib/server/venv.js`; `prepack`/`prepublishOnly` run `vite build`); launcher defaults to the prod build (`node build/index.js`), `--dev` uses Vite. `build/`/`.venv/` gitignored. A root **`Makefile`** wraps these mechanics for convenience (`make setup` = install+build+`chmod +x bin/cellar.js`+`npm link`; `make update` = `git pull`+install+build; plus `make build`/`run`/`dev`; bare `make` prints help) - it invents no new scripts, just the real npm/launcher commands. `bin/cellar.js` is committed **executable** (git mode 100755 - the shebang'd, sole `package.json` `bin` entry); preserve that bit if you ever regenerate it, or `npm link` yields a non-runnable `cellar` symlink (the `make setup` `chmod` is a belt-and-suspenders re-set).
- **Per-repo launcher + venv binding (`bin/cellar.js` + `src/lib/server/venv.js`).** `cellar` in any repo binds the kernel to the **project's** venv. Resolution (first match): `--venv`/`CELLAR_VENV` → `$VIRTUAL_ENV` → `<ws>/.venv` → create. All venv create/installs go through **`uv`** (`uv venv`, `uv pip install`) — hard requirement, fails fast if absent (no `python -m venv` fallback). Ensures `ipykernel` in the project venv; **confirms before** create/install on a TTY (`--yes`/`-y`/`$CI`/non-TTY auto-approve; `--python` = bind arbitrary interpreter, no create/install). `venv.js` uses node builtins only so it's importable by both the launcher (`../src/lib/server/venv.js`) and the SvelteKit server (`$lib`). Cellar's heavy Jupyter host env (`jupyter-server`) lives in a private `~/.cellar/host-venv`, created/cached first run, kept separate from project venvs.
- **Kernel bridge = the de-risked core.** `src/lib/server/kernel.js` connects to Jupyter over REST+WebSocket using `@jupyterlab/services` (spec §2). Node 18+ global `fetch`/`WebSocket` are passed into `ServerConnection.makeSettings` — no polyfill. `execute()` emits real nbformat output objects so the same shape streams live to the browser and is persisted. `rebindKernel()` (added) tears down + restarts the kernel so it re-reads a rewritten kernelspec onto a new interpreter (a plain `restart()` reuses the original argv and would NOT switch venvs).
- **Kernelspec = the venv lever.** The launcher writes a per-run `python3` kernelspec (`writeKernelspec`) whose `argv[0]` is the project python, into a temp `JUPYTER_PATH` dir handed to the sidecar; `startNew({name:'python3'})` then launches the project interpreter — `kernel.js` unchanged. The dir is passed to the app as `CELLAR_KERNELSPEC_DIR` so the runtime venv control can rewrite it.
- **Runtime venv control.** Settings → "Python environment" (`src/lib/Settings.svelte`) → `src/routes/api/venv` (GET info / POST `{path,create}`) → `src/lib/server/venv-bind.js`: resolves/creates the venv via uv, ensures ipykernel, rewrites the kernelspec, updates in-process `CELLAR_PROJECT_VENV`, then `rebindKernel()`. Confirm-before-create is enforced in the UI.
- **Notebook documents (server-owned, multi-doc).** `src/lib/server/notebook.js` holds in-memory notebooks keyed by absolute path in a `docs` Map — the default `notebook.ipynb` **and** any other workspace `.ipynb` opened live. Every exported op takes an optional trailing `nb` (workspace-relative or absolute path); omitting it targets the **active** notebook (`activePath`). `setActiveNotebook(nb)`/`getActiveNotebookPath()` track which notebook the agent-facing (MCP) tools default to; the UI POSTs `/api/notebooks` to follow tab focus. The default notebook is created if missing + re-persisted (normalized) on load; an arbitrary opened `.ipynb` must exist and is NOT rewritten until actually mutated (no surprise git diff on open). Cellar owns cell IDs (uniqueness enforced **per document** on load via `enforceUniqueIds`; ids may legitimately repeat across notebooks — do NOT assume global uniqueness).
- **Persistence + clean-on-save.** `src/lib/server/ipynb.js` (de)serializes canonical doc ↔ real nbformat 4.5 with **deterministic** output (fixed key order, 1-space indent) so identical re-runs produce zero git diff. `src/lib/server/clean.js` is the nbdev field-policy port: nulls all `execution_count`, deny-by-default metadata allowlist (cell keeps only the `cellar` namespace; notebook keeps only `kernelspec` — drops `language_info`/`widgets`), normalizes `kernelspec.display_name`→`name`, scrubs `<… at 0x…>` memory addresses. Must stay **idempotent**.
- **API.** `+page.server.js` load returns the default notebook + seeds the sidebar. `src/routes/api/notebooks` (`GET ?path=` loads a live doc's cells; `POST {path}` sets the active notebook). Cell ops under `src/routes/api/cells/…` (`POST` add, `PATCH` source/type, `DELETE`, `POST …/run` streams NDJSON + persists, `…/move`, `…/clear`) all carry an `nb` path (body field, or `?nb=` query for clear/delete) so they mutate the addressed notebook, not just the active one.
- **Cell types.** Code and **Markdown** (nbformat `cell_type`). Markdown cells carry source + no outputs/execution_count; clean-on-save + UUID ids apply the same. `addCell(afterId, cellType)` and `setCellType(id, type)` (PATCH `cell_type`); converting to markdown clears outputs.
- **UI.** `src/lib/Cell.svelte` = one cell — CodeMirror editor (language via a Compartment: python / markdown; editor theme via a second Compartment that follows the app theme; `theme` prop drilled shell→`LiveNotebook`→`Notebook`→`Cell` (and shell→`FileTab`)). **Editor theming is centralized in `src/lib/editorTheme.js`** (`editorThemeExtensions(theme)` + `isLightTheme(theme)`), shared by `Cell.svelte` and `FileTab.svelte`: light vs dark is derived from the **resolved `color-scheme`** daisyUI sets on `<html data-theme>` (light themes get the pygments-"default" Jupyter light scheme, dark get `oneDark`), not a hardcoded theme-name allowlist, so every light theme is handled automatically (the `LIGHT_THEMES` name set is only an SSR/pre-DOM fallback). The reconfigure `$effect` must read `theme` **outside** the `if (view)` guard - `view` is assigned in `onMount`, which can run after the effect's first pass, so a guarded read never registers `theme` as a dependency and the editor stays stuck at its mount-time theme on toggle. `Cell.svelte` also carries the per-cell toolbar and renders nbformat outputs. Markdown cells render to sanitized HTML (markdown-it `html:false` + DOMPurify, client-only via `browser` guard); double-click / pencil button → raw edit; Run/⌘Enter renders in place, Shift+Enter renders+advances. Type toggle is the far-right `python3`/`markdown` label. Per-notebook cell ops (add code/markdown, delete/reorder, single-kernel one-run-at-a-time; markdown "run" persists source only, no kernel) live in `src/lib/LiveNotebook.svelte`, which owns one notebook's cells + renders via `Notebook.svelte` (pure renderer). `Cell` seeds its editor doc once on mount and does NOT sync later `cell.source` changes into the editor — so `LiveNotebook` fetches authoritative cells on mount (never pre-seeds a stale snapshot). Editor source autosaves via debounced `PATCH`. Dev exposes `window.cellarViews[id]`. **Active cell** = `LiveNotebook` `activeId` (set on a Cell's `onfocusin`/`onpointerdown`, and on keyboard move); the active Cell gets a `border-primary` ring + a left accent bar (`data-active`), works in both themes. **Drag-to-reorder cells**: a per-cell grip handle (`data-testid=drag-handle`, `draggable`) + HTML5 DnD drop zones in `Notebook.svelte` → `onMoveToIndex` → the move route's `{toIndex}` → server `moveCellTo` (persists + git-clean like any move; keyboard ⌘/Ctrl+Shift+Arrow still works). **Scrollable outputs** (Jupyter "Enable Scrolling for Outputs"): a per-cell toggle contracts a tall output to a fixed-height scroll box; auto-applies above `SCROLL_THRESHOLD` px unless overridden. The choice persists in `metadata.cellar.output_scrolled` (undefined=auto / true / false) — round-trips through clean-on-save; note `cellView` in `notebook.js` now carries `metadata` so the browser sees it, set via PATCH `{scrolled}`. **Markdown tables in text/stream output** are parsed (header + `|---|` separator) and rendered as sanitized daisyUI `table table-zebra table-xs` (same markdown-it + DOMPurify path as markdown cells); non-table text is unchanged.
- **MCP agent interface.** `src/lib/server/mcp/` — the agent surface (spec §4). `service.js` = transport-independent tool logic (section tree from markdown headers, tiered/summarized outputs, visibility filtering, UUID-addressed read/write/execute); `server.js` = `McpServer` + Streamable HTTP transport on its own port (`CELLAR_MCP_PORT`, default 39587 but the launcher now allocates a **free** port per instance so concurrent `cellar` runs don't collide, path `/mcp`), started once from `src/hooks.server.js`. Runs **in-process** in the SvelteKit backend so it shares the live `notebook.js` doc + `kernel.js` — restarting the kernel can't drop the MCP session/document (the #1 requirement). Per-cell hide/show via `metadata.cellar.hidden_from_agent` (allowlisted, visible by default) is honored in every map/read/search/section/result. Kernel service methods `restartKernel/interruptKernel/kernelStatus` live in `kernel.js`; richer read/write (`listCells/getCell/setVisibility/moveCellTo`) in `notebook.js`. **Coherence doctrine:** `server.js` passes an `INSTRUCTIONS` house-style string as the `McpServer` `instructions` (advisory, delivered once at connect: imports-at-top convention, check `kernel_state` first, continue the story via `get_notebook_map`, structure with markdown) + a matching `cellar_notebook_style` prompt. The read-group **`kernel_state`** tool returns the live namespace bucketed into imports/functions/classes/variables (types, shapes, dataframe columns); it reuses `inspect.js`'s shared probe (`kernelState()`), short-circuits to `{started:false}` when `kernelStatus()` is `not_started` (no force-boot) and to `{started:true,busy:true}` when the kernel is busy (never queues a probe behind a running cell), and dedupes imports by statement. `inspectVariables()` (sidebar) shares that probe but projects to the original `{name,type,shape,preview}` shape.
- **Zero-config agent connection (`cellar mcp`).** Because the MCP HTTP port is per-run dynamic, agents are pointed at the stdio command `cellar mcp`, never a URL. On launch `bin/cellar.js` writes `<ws>/.cellar/runtime.json` (`{mcpPort,appPort,jupyterPort,pid}`, ephemeral - gitignored, cleared in `cleanup()` on shutdown) and writes/merges `<ws>/.mcp.json` with a `cellar` stdio entry (`--no-mcp-config` opts out). Both helpers live in `src/lib/server/runtime.js` (node-builtins + `fetch` only, importable by launcher and `$lib`); `.mcp.json` merge is idempotent and preserves other servers. The `mcp` subcommand (handled at the top of `bin/cellar.js`, before normal arg parsing) calls `runMcpBridge` in `src/lib/server/mcp-bridge.js`: reads runtime.json, verifies alive (pid AND a GET to `/mcp`), then transparently proxies JSON-RPC both ways between a `StdioServerTransport` and a `StreamableHTTPClientTransport` to the live server (schema-agnostic, so it never drifts). Missing/dead instance → clear stderr error + exit 1 (never auto-launches). `runMcpBridge` returns a promise that stays pending until shutdown - the launcher must `await` it (an early `process.exit` kills the bridge right after connect). Both new modules are added to `package.json` `files`. The sidebar "Connect an agent" panel is reframed around this flow (see Sidebar entry below).
- **Not yet built (deferred by design):** MCP `get_changes_since`/delta reads + subscribable-resource map, extract-to-`.py`, `.py` view, Databricks, git merge driver, per-output size threshold, LaTeX/math + code-block highlighting in rendered markdown, polished editing UX.

## Application shell (navbar + sidebar)

Wraps the notebook in a JupyterLab-style shell. `src/routes/+page.svelte` is the shell: it owns tabs/sidebar/kernel-controls/theme and hosts one `LiveNotebook` per open notebook tab (default + each opened `.ipynb`), all kept mounted+hidden so editor/run state survives tab switches. It no longer owns `cells`: each `LiveNotebook` owns its own cells and reports the live array up via `onCellsChange(path, cells)` into a path-keyed `notebooksCells` map; the sidebar (outline/search/variables) reads `activeCells` for the focused notebook (falls back to the default when a plain file tab is active). A single shared kernel → the shell serializes runs across notebooks via a global `runBusy` flag (`onRunStart`/`onRunEnd`).

- **Navbar** (`src/lib/Navbar.svelte`): app-menu dropdown → `src/lib/Settings.svelte` modal (theme toggle `dim`↔`nord`, persisted to `localStorage['cellar-theme']`, applied via `<html data-theme>`); editor-style **wrapping** tab bar (tabs flow onto more rows via `flex-wrap`, never clip). Every tab (including the notebook) is closable and part of session memory; tabs carry file-type icons + a dirty dot; `preview` tabs render italic.
- **File tabs** (`src/lib/FileTab.svelte`): CodeMirror editor over a workspace file, ⌘S save; syntax-highlighted by file extension (`langFor`: python / markdown / json+ipynb / yaml, else plain text), sharing `editorTheme.js` light/dark theming with cells; dirty state bubbles up to the tab bar. Notebook + all file tabs stay mounted (hidden via `.hidden`) so editor/run state survives tab switches.
- **Sidebar** (`src/lib/Sidebar.svelte`, foldable sections; whole sidebar collapsible + **drag-resizable** via the `sidebar-resizer` handle in `+page.svelte`, width persisted to `localStorage['cellar-sidebar-width']`, bounds 180–560px): **Files** (`src/lib/FileTreeNode.svelte`, recursive), **Kernels** (attached notebook; the attached row opens the live notebook tab), **Connect an agent** (`agentSection` in `Sidebar.svelte`, reframed around the zero-config flow: leads with an agent-ready banner when a project `.mcp.json` registers the `cellar` stdio server — detected server-side in `+page.server.js` load via `mcpConfigPath`, surfaced as `data.mcp.projectConfigured`, with a `--no-mcp-config` fallback message; shows the one-time `claude mcp add cellar -- cellar mcp` command with a copy button; demotes the live raw `/mcp` Streamable-HTTP URL + config snippet under an "Advanced · raw endpoint" disclosure. The raw URL still comes from `data.mcp.url` reading `CELLAR_MCP_PORT`), **Outline** (nested markdown-header tree; a node with children collapses via a per-node chevron, state keyed by heading path in `localStorage['cellar-outline-collapsed']`), **Variables**, **Search**. Sections render from a persisted order array (`localStorage['cellar-sidebar-order']`) and are **drag-reorderable** via a per-section grip handle (`data-testid=section-drag-<key>`, native HTML5 DnD, no library); each section's open/closed state persists to `localStorage['cellar-sidebar-open']`. Section bodies are Svelte `{#snippet}`s dispatched by key so order is data-driven.
- **Shell backend.** `src/lib/server/fstree.js` + `src/routes/api/fs/{tree,file}` — workspace listing and file read/write, all path-guarded to the workspace root. `src/lib/server/inspect.js` + `src/routes/api/kernel/variables` — variable inspector; runs a Python probe through `execute()` (starts from IPython `user_ns` minus `user_ns_hidden`, drops modules/functions/classes + typing/magics scaffolding). `src/routes/api/kernel` + `getKernelInfo()` (added to `kernel.js`) — read-only kernel status that never boots a kernel. Variables refresh after each run and on mount (only if a kernel already exists).
- **Kernel controls.** Sidebar Kernels section has **Interrupt**/**Restart** buttons (`data-testid=kernel-interrupt`/`kernel-restart`), disabled until the kernel is started. They POST `src/routes/api/kernel/{interrupt,restart}`, which call `interruptKernel()`/`restartKernel()` in `kernel.js` — the same restart-over-same-session path the MCP interface uses, so the doc survives while the namespace clears (the page drops stale inspector rows on restart).
- **Move-cell shortcuts.** `LiveNotebook.svelte` `onKeydown` (window capture-phase, registered only while that notebook is the active tab, so it beats CodeMirror + the browser): ⌘/Ctrl+Shift+Arrow moves the focused cell (`closest('[data-cell-id]')`) via `moveCell`, then restores focus through the `focusers` registry so moves chain; no-ops at the edges.
- **Fluid notebook width.** `Notebook.svelte` content column is `max-w-[clamp(48rem,92%,88rem)]` — grows with the viewport up to a readable 88rem cap (not full-bleed on ultrawide).
- **File tree UX (VS Code-style).** Folders start **collapsed** (`FileTreeNode` `open=false`), only root visible. Entries carry self-contained inline-SVG **file-type icons** (`src/lib/fileIcons.js`, keyed by extension, no CDN/network — offline-safe). **Git decorations**: `src/lib/server/git.js` + `src/routes/api/fs/git` return `git status --porcelain -uall` as `{isRepo, files:{relPath→letter}}`; the tree colors names + shows a status letter (M gold, A/U green, D/C red) and rolls child status up to collapsed folders (a dot). Refreshed on mount, manual refresh, window focus, and after file saves (`fsRefreshSignal` bump); a non-git workspace returns `isRepo:false` → clean tree, no errors.
- **Tabs — session memory + preview.** Open-tab set is persisted per workspace to `localStorage['cellar-tabs:'+workspace]` (paths + preview flag + active id). **First-ever open** (no key) → empty state (`data-testid=empty-state`, no tabs); returning → tabs restored. Single-click a tree file → transient **preview** tab (one at a time; next single-click reuses its slot); **double-click** (tree or tab) or **editing** promotes it to permanent (VS Code semantics). All tab logic lives in `+page.svelte`.
- **Opening a workspace `.ipynb` (live).** Any `.ipynb` opened from the tree loads as a **live, kernel-attached** notebook via `LiveNotebook` (view/edit/run, edits saved back to that file through clean-on-save) — `ipynb`-kind tabs render `LiveNotebook path={tab.path}`, the canonical one uses tab id `notebook`. Preview/permanent tab semantics are unchanged (single-click preview slot, double-click/edit promotes). The old read-only `NotebookFileView` + `/api/fs/notebook` + `readWorkspaceNotebook` were removed.
