## Current state: MVP Step 1 — notebook document + save pipeline

Built on the SvelteKit ↔ Jupyter kernel bridge. Full product/tech spec lives outside the repo at `firstmate/data/cellar-spec.md` (read §3 for the versioning model); nbdev field-policy detail at `firstmate/data/nbdev-study-n8/report.md`. See `README.md` for run/verify details.

- **Two runtimes.** Node (SvelteKit) + a Python sidecar (Jupyter kernel service). Both are booted by the CLI launcher `bin/cellar.js` (one command → both servers up → browser opens, workspace = cwd). Distributed as an **npm package** (`package.json` `files`=`build/`+`bin/`+`src/lib/server/venv.js`; `prepack`/`prepublishOnly` run `vite build`); launcher defaults to the prod build (`node build/index.js`), `--dev` uses Vite. `build/`/`.venv/` gitignored. A root **`Makefile`** wraps these mechanics for convenience (`make setup` = install+build+`chmod +x bin/cellar.js`+`npm link`; `make update` = `git pull`+install+build; plus `make build`/`run`/`dev`; bare `make` prints help) - it invents no new scripts, just the real npm/launcher commands. `bin/cellar.js` is committed **executable** (git mode 100755 - the shebang'd, sole `package.json` `bin` entry); preserve that bit if you ever regenerate it, or `npm link` yields a non-runnable `cellar` symlink (the `make setup` `chmod` is a belt-and-suspenders re-set).
- **Per-repo launcher + venv binding (`bin/cellar.js` + `src/lib/server/venv.js`).** `cellar` in any repo binds the kernel to the **project's** venv. Resolution (first match): `--venv`/`CELLAR_VENV` → `$VIRTUAL_ENV` → `<ws>/.venv` → create. All venv create/installs go through **`uv`** (`uv venv`, `uv pip install`) — hard requirement, fails fast if absent (no `python -m venv` fallback). Ensures `ipykernel` in the project venv; **confirms before** create/install on a TTY (`--yes`/`-y`/`$CI`/non-TTY auto-approve; `--python` = bind arbitrary interpreter, no create/install). `venv.js` uses node builtins only so it's importable by both the launcher (`../src/lib/server/venv.js`) and the SvelteKit server (`$lib`). Cellar's heavy Jupyter host env (`jupyter-server`) lives in a private `~/.cellar/host-venv`, created/cached first run, kept separate from project venvs.
- **Kernel bridge = the de-risked core.** `src/lib/server/kernel.js` connects to Jupyter over REST+WebSocket using `@jupyterlab/services` (spec §2). Node 18+ global `fetch`/`WebSocket` are passed into `ServerConnection.makeSettings` — no polyfill. `execute()` emits real nbformat output objects so the same shape streams live to the browser and is persisted. `rebindKernel()` (added) tears down + restarts the kernel so it re-reads a rewritten kernelspec onto a new interpreter (a plain `restart()` reuses the original argv and would NOT switch venvs). **Startup injection (`initKernel`):** every kernel silently runs `%matplotlib inline` on fresh start **and** after `restartKernel()` (restart clears the namespace + backend), so matplotlib figures emit `image/png` even when the project's default backend isn't inline (e.g. `MPLBACKEND=agg`/macosx); guarded by try/except so it's a harmless no-op when matplotlib is absent.
- **Run watchdog = liveness PROBE, never silence (`kernel.ts` `execute()`).** `await future.done` is unbounded (a stalled websocket / an undelivered `execute_reply` / a dropped autorestart wedges the queue slot forever), so a watchdog guards it - but **silence is NOT death, and must never abort a run.** A Spark query (`sql.ts` compiles to `spark.sql(...).limit(N).toPandas()`), a big pandas op, or untraced training is ONE blocking call emitting NOTHING until it returns; the old abort-on-idle-window watchdog could not tell a 20-minute cluster job from a dead kernel and killed the user's real work ("Kernel went unresponsive (no activity for 900s)" on a healthy kernel). So `CELLAR_KERNEL_IDLE_TIMEOUT_MS` is now a **probe interval, not a deadline** (default **30s**, was 15min - a probe is cheap and no longer destructive, so a wedged notebook recovers in one window (proven death) to three (`SUSPECT_STRIKES`), ~30-90s, instead of waiting 15min for the watchdog's one shot). Expiry only runs `probeKernelLiveness`; **only its verdict may abort**. Any kernel traffic still re-arms, so a cell that emits output is never probed at all. **`CELLAR_KERNEL_IDLE_TIMEOUT_MS=0` disables the per-run watchdog entirely** (`kernelIdleTimeoutMs` maps an explicit `0` to the `WATCHDOG_DISABLED` sentinel; `armWatchdog` never arms → never probes, never aborts, run relies only on the force-abort path), matching the culler's `CELLAR_KERNEL_IDLE_TIMEOUT=0`; empty/unset/non-numeric/negative still fall back to 30s via `envMs`, and the probe TIMEOUT knob keeps rejecting non-positive values as typos.
  - **The one out-of-band signal is the sidecar's REST view - `GET /api/kernels/<id>` (`KernelAPI.getKernelModel`).** Empirically verified against a real sidecar running a blocking `time.sleep`: it answers in **~3-5ms with `execution_state:'busy'` while the shell channel is blocked**, because it is the Jupyter SERVER's own bookkeeping. **What does NOT work, and why (do not retry these):** a `kernel_info_request` - or ANY `execute`, **`{internal:true}` included** (that bypasses Cellar's run queue, NOT the busy kernel) - rides the **shell channel** and QUEUES behind the running cell, so it answers only once the cell ends and would read "unresponsive" for exactly the workload this protects; and `getKernelInfo()`/`listKernels()` report the **local** `connection.status`, which a stalled websocket freezes at `busy` forever. A gone kernel surfaces as `getKernelModel` resolving **`undefined`** (it resolves the 404, it does not throw); a throw means the sidecar itself was unreachable.
  - **Abort rule:** Gone-from-server (`undefined`) or `execution_state:'dead'` ⇒ **proven dead**, abort immediately (not a race that resolves) - the REST verdict outranks any socket state, since a kernel the server no longer has is dead whatever the socket says. Else the server still HAS the kernel, so the verdict turns on whether we can still HEAR it, and **exactly two readings may accrue a strike**: (a) `connectionStatus==='disconnected'` ⇒ **suspect** - @jupyterlab exhausted its 7 retries, so the reply can never arrive whatever the kernel's state; (b) `connectionStatus==='connected'` **and** `execution_state!=='busy'` ⇒ **suspect** - we are connected, so we would have heard the reply, and the kernel is not running our cell: the genuine lost-reply case, the transport failure this exists for. Connected **and** `busy` ⇒ **alive**, re-arm (repeats indefinitely - a 3-hour job survives). Any other socket state - in practice **`'connecting'`**, @jupyterlab mid-reconnect - ⇒ **unknown/inconclusive, re-arm, NEVER a strike, regardless of `execution_state`**. That last rule is load-bearing in BOTH of the blip's shapes: `busy` is proof the kernel works, and `idle` does **not** mean the reply was lost - a cell that FINISHED mid-blip leaves the kernel idle with its `execute_reply`/iopub **buffered** for replay (`buffer_offline_messages`, default on), so convicting there would destroy a result already computed and report a loss that is about to arrive. Either way it would abort exactly the silent multi-hour Spark job this exists to protect. A truly dead socket is not spared, only slower: @jupyterlab exhausts its retries (~120s) into `'disconnected'`, which convicts via (a)'s strike path - so dead-socket recovery costs **~120s + the strike window (~90s at the default) ≈ 210s, delayed but never lost** - the explicitly accepted price. A suspect verdict aborts only after `SUSPECT_STRIKES` (3) **consecutive** readings, like `parent-watch`'s CONFIRM_STRIKES. Probe threw **or timed out** ⇒ the one reading with no view of the kernel at all, so it turns entirely on the socket (`probeFailureVerdict`): with the socket `'connected'`/`'connecting'` it is **inconclusive, NEVER abort** - the instance-reaper's stance (`pidReapDecision`): killing REQUIRES a positive match, because the costs are asymmetric (a false abort destroys an hour-old cluster job; a false re-arm only delays recovery of a notebook the user can restart by hand). That is what keeps a silent Spark query safe through a flaky sidecar: its socket is connected, so failing probes alone can never convict it. **EXCEPT** a failed/timed-out probe **and** `connectionStatus==='disconnected'` ⇒ **suspect** (the third strike-accruing reading, converting through the normal strike path): the websocket rides the SAME sidecar, so HTTP failing and the socket having exhausted its retries are two INDEPENDENT, CORROBORATING signals that we cannot hear this kernel by ANY route - a positive match, which UPHOLDS the doctrine rather than weakening it. Without that carve-out an unconditional `unknown` reset the strikes forever, so a **persistently** wedged sidecar (HTTP black-holed while the TCP socket stays up) wedged the queue slot PERMANENTLY - the exact failure the watchdog exists to clear, and one the old 15-minute abort did free. The inconclusive stance is for a **transient** failure only. Its message obeys the same honesty rule and is the strictest case: having lost BOTH routes we know LESS about the cell than any other path, so it says only that the kernel is unreachable by any route and the run was abandoned - never that the cell stopped. (Deliberately NOT implemented: preserving strikes across `unknown`, which would let a purely transient flake accumulate toward conviction.) Aborts keep the same `KernelExecuteAborted`/`'idle_watchdog'` reason + single-release path, but the message now names the real cause, not "no activity for Ns". **Each verdict carries its OWN complete sentence** (`ProbeResult.message`) - never a shared prefix in front of a fragment, because the paths know different things: gone/dead and connected-but-not-busy may say the kernel is no longer running the cell (the server lost it / we are connected so we would have heard the reply), but **`'disconnected'` may NOT** - without the socket we cannot see whether the cell is still running, and on a healthy kernel it very likely is, so it says only that the connection was lost and the run abandoned. Asserting a cause a reading did not establish is the exact defect ("no activity for 900s") this whole watchdog exists to retire; don't reintroduce it on the one path least able to know. A kernel PROCESS death is not this probe's case: jupyter **autorestarts** it and `abortActiveRuns(…,'kernel_autorestart')` already settles the run. Covered by `tests/unit/kernel-execute-watchdog.test.ts` + `tests/e2e/kernel-watchdog-probe.spec.ts`.
  - **The probe MUST always settle, or it disarms the watchdog it drives.** `armWatchdog()` only re-arms *after* a probe settles, so a probe that never resolves is strictly worse than one that aborts wrongly: the watchdog stops cycling for the rest of the run and the wedged queue slot it exists to free stays wedged forever. `fetch` has **no default request timeout**, so a sidecar that accepts the connection and then says nothing (black-holed socket, wedged jupyter-server) hits exactly that. So the probe threads an `AbortSignal` through `makeSettings(signal)` (`ServerConnection.ISettings.init`, which @jupyterlab merges into the request as `{...settings.init, ...init}` - note it **replaces** the library's default `init`, hence `cache`/`credentials` are restated there) and races it against `probeTimeoutMs()` (**10s**, `CELLAR_KERNEL_PROBE_TIMEOUT_MS`), so the hung request is **cancelled**, not merely ignored and left leaking a socket. That controller is **hoisted into `execute()`**, so the run's `finally` also cancels a probe still in flight when the run settles (its verdict is already discarded by the settled/aborted guard) rather than letting a hung GET hold its socket for the rest of the timeout. The timeout is a **fixed constant, deliberately NOT derived from `idleMs`** - tests drive that window to 40ms and would time out every probe. At the **30s default window** it cannot cause overlapping probes: `armWatchdog()` re-arms only after a probe settles, and the only other arming path (`resetWatchdog`, on kernel traffic) cannot fire a second probe inside a window longer than the timeout. Probes are **not** sequential by construction, though - drive the window below the probe timeout and traffic mid-probe can start a second one; that is tolerable (and deliberately carries no overlap guard) because a verdict only ever re-arms or accrues a strike. A timed-out probe is `unknown` (re-arm, strikes reset) while the socket is still connected/reconnecting, **never** an abort - a hung sidecar alone proves nothing about the kernel, and the point is only that the watchdog keeps CYCLING so it can still convict on a later real dead/suspect verdict (or once the socket itself gives up: see the corroborated carve-out in the abort rule above).
- **`%restart_python` magic + comm subshells (`src/lib/server/controlMagic.ts`).** `initKernel` also injects (via `runSilent`, every start + restart) a `%restart_python` line magic (Databricks-style fresh-namespace restart). It's a **managed** restart, not a self-kill: the magic prints "Restarting Python kernel..." and opens a `cellar.control` comm with `{op:'restart'}`; `kernel.ts` `registerControlComm` catches it and runs `restartKernel(nbPath)` for THAT notebook only (epoch bumps → cells read not-run-this-session; widgets/queue cleared; `sys.path` project-root re-applied; sidebar card kept). The restart is **deferred until the kernel is next idle** (`whenIdle`) because the `comm_open` lands mid-run and tearing the process down before the cell's `execute_reply` would reject that run's future. **Load-bearing:** `getKernel` sets `kernel.commsOverSubshells = CommsOverSubshells.Disabled` before any comm opens. @jupyterlab/services' default under ipykernel 7 is `PerCommTarget`, which issues a `create_subshell_request` per comm target; `restart()` cancels that future and @jupyterlab rejects it with NO awaiter → an **unhandled rejection that crashes the Node app**. `%restart_python` (comm-then-restart) hit this every time, and a widget open racing a restart is the same latent trap; Cellar serializes runs so subshells buy nothing. Covered by `tests/unit/control-magic.test.ts`. **`%run` is IPython's built-in** and works unchanged: the kernel's cwd is WORKSPACE (sidecar spawn), so `%run helper.py` / `%run subdir/x.py` resolve workspace-relative; no Cellar code needed.
- **Packaging + self-update (`cellar --version` / `cellar --update`).** Both flags are handled at the top of `bin/cellar.js` (right after `mcp`, before normal arg parsing) so they never boot a server, and delegate to `src/lib/server/selfupdate.js` (node-builtins only, in `package.json` `files`). `--version` prints package.json version + git short-sha/branch/date/dirty (of `REPO`, the launcher's own dir) + detected install method. `--update` is **install-method aware** (`detectInstall`): Homebrew (running path under `brew --prefix`/a `/Cellar/cellar/` keg, or `brew list cellar` succeeds) → `brew update && brew upgrade --fetch-HEAD cellar` (the `--fetch-HEAD` is load-bearing — a HEAD keg won't move without it); else a git clone → `git pull --ff-only` + `npm ci`/`npm install` + `npm run build` (the Makefile `update` steps). Cellar tracks **latest** (git `main`), not semver. **Homebrew tap:** a HEAD-only formula (canonical copy `packaging/homebrew/cellar.rb`; installable copy in the private tap repo `fbereilh/homebrew-cellar` at `Formula/cellar.rb` — keep in sync) `depends_on "node"`+`"uv"`, `npm ci && npm run build`, installs into `libexec`, `bin/cellar` wrapper runs `node libexec/bin/cellar.js`. Audits clean under `brew audit --strict --formula` / `brew style`. Private repo → `brew install --HEAD` needs the user's git creds.
- **Kernelspec = the venv lever.** The launcher writes a per-run `python3` kernelspec (`writeKernelspec`) whose `argv[0]` is the project python, into a temp `JUPYTER_PATH` dir handed to the sidecar; `startNew({name:'python3'})` then launches the project interpreter — `kernel.js` unchanged. The dir is passed to the app as `CELLAR_KERNELSPEC_DIR` so the runtime venv control can rewrite it.
- **Sidecar cwd must equal `root_dir` (= WORKSPACE).** `bin/cellar.js` spawns the Jupyter sidecar with `cwd: WORKSPACE`, not Cellar's `REPO`. Because `startNew({name:'python3'})` passes no notebook path, `jupyter_server` doesn't derive a kernel cwd from `root_dir` and the kernel inherits the sidecar's process cwd — so anchoring it at WORKSPACE is what makes `os.getcwd()`, relative reads/writes, and repo-root walks resolve in the user's project (else they resolve in / write into Cellar's install dir). Everything else in that spawn (host python, `JUPYTER_PATH`) is already an absolute path, so it stays REPO-independent. **Follow-up (not done):** for multi-directory workspaces, pass each notebook's workspace-relative path to `startNew(...)` so a kernel roots at its own notebook's folder — deferred because a single kernel is shared across all open notebooks, making a per-notebook root ambiguous.
- **Runtime venv control.** Settings → "Python environment" (`src/lib/Settings.svelte`) → `src/routes/api/venv` (GET info / POST `{path,create}`) → `src/lib/server/venv-bind.js`: resolves/creates the venv via uv, ensures ipykernel, rewrites the kernelspec, updates in-process `CELLAR_PROJECT_VENV`, then `rebindKernel()`. Confirm-before-create is enforced in the UI.
- **Notebook documents (server-owned, multi-doc).** `src/lib/server/notebook.js` holds in-memory notebooks keyed by absolute path in a `docs` Map — the default `notebook.ipynb` **and** any other workspace `.ipynb` opened live. Every exported op takes an optional trailing `nb` (workspace-relative or absolute path); omitting it targets the **active** notebook (`activePath`). `setActiveNotebook(nb)`/`getActiveNotebookPath()` track which notebook the agent-facing (MCP) tools default to; the UI POSTs `/api/notebooks` to follow tab focus. **Loading never writes to disk** (`loadDoc`): the default notebook is materialized *in memory* if missing so callers get a valid shape (SSR seeds the shell from `getDefaultNotebook()`), but no `.ipynb` is written on a bare startup/open — so opening Cellar in a folder drops **no** uninvited file. Every notebook (default included) persists only on a genuine mutation (add/edit/run/…) or an explicit `createNotebook`, which materializes the file (and the UI's empty-state "New notebook" button → `POST /api/notebooks {create:true}` → `createNotebook`); an arbitrary opened `.ipynb` must exist and is likewise NOT rewritten until mutated (no surprise git diff on open). Startup with no `notebook.ipynb` shows the shell's empty state (`+page.svelte`, `data-testid=empty-state`). Cellar owns cell IDs (uniqueness enforced **per document** on load via `enforceUniqueIds`; ids may legitimately repeat across notebooks — do NOT assume global uniqueness).
- **Persistence + clean-on-save.** `src/lib/server/ipynb.js` (de)serializes canonical doc ↔ real nbformat 4.5 with **deterministic** output (fixed key order, 1-space indent) so identical re-runs produce zero git diff. `src/lib/server/clean.js` is the nbdev field-policy port: nulls all `execution_count`, deny-by-default metadata allowlist (cell keeps only the `cellar` namespace; notebook keeps only `kernelspec` — drops `language_info`/`widgets`), normalizes `kernelspec.display_name`→`name`, scrubs `<… at 0x…>` memory addresses. Must stay **idempotent**. **Writes are atomic + serialized (the notebook is the user's PRIMARY data):** `writeNotebook` snapshots the cleaned JSON synchronously, then persists via `src/lib/server/atomic-write.ts` `atomicWriteFileSync` — write a **unique** temp *in the target's own dir* (a cross-device `/tmp` rename is NOT atomic), `fsync`, then `renameSync` over the target; on any error the temp is removed and the original is left untouched, so a crash/disk-full/SIGKILL mid-write can never truncate the `.ipynb` (a reader sees only the complete old or complete new file). Persist stays **synchronous** (the on-disk file must exist the instant persist returns — the unload flush and the git-blame/diff paths read it right after writing), so `src/lib/server/write-lock.ts` `serializeWriteSync(resolvedPath, fn)` runs inline when the path is idle (the normal case — a synchronous critical section can't be interrupted) and only queues behind an in-flight **async** writer; `withPathLock` is the async `Map<path,Promise>` tail-chain for that case, with the map entry deleted when a path's chain drains (never grows). Unique temp names mean even two racing writes to the same path each own a private temp — neither observes the other's half-written bytes; last complete rename wins. `.py` (jupytext) writes go through the external jupytext writer (`writePyNotebook`), unchanged. Covered by `tests/unit/atomic-write.test.ts`.
- **API.** `+page.server.js` load returns the default notebook + seeds the sidebar. `src/routes/api/notebooks` (`GET ?path=` loads a live doc's cells; `POST {path}` sets the active notebook, or `POST {path,create:true}` materializes it on disk via `createNotebook`). Cell ops under `src/routes/api/cells/…` (`POST` add, `PATCH` source/type, `DELETE`, `POST …/run` streams NDJSON + persists, `…/move`, `…/clear`) all carry an `nb` path (body field, or `?nb=` query for clear/delete) so they mutate the addressed notebook, not just the active one.
- **Cell types.** Code and **Markdown** (nbformat `cell_type`). Markdown cells carry source + no outputs/execution_count; clean-on-save + UUID ids apply the same. `addCell(afterId, cellType)` and `setCellType(id, type)` (PATCH `cell_type`); converting to markdown clears outputs.
- **UI.** `src/lib/Cell.svelte` = one cell — CodeMirror editor (language via a Compartment: python / markdown). **Editor theming is centralized in `src/lib/editorTheme.js`, which exports one constant extension `EDITOR_THEME`** shared by `Cell.svelte` and `FileTab.svelte`. It is deliberately **theme-independent**: every color is a `--cellar-cm-*` custom property resolved in `app.css` by `light-dark()` against the **resolved `color-scheme`** daisyUI sets on `<html data-theme>` (never a theme-name allowlist), so a theme toggle is a plain CSS repaint that dispatches **nothing** into any editor. **Do not reintroduce a theme `Compartment`**: swapping a theme through one makes CodeMirror re-mount the single shared `<style>` element it keeps in `document.head`, invalidating document-wide style, and doing that once per mounted editor turned a toggle into N full-document style+layout recalcs (~16ms each → ~800ms of blocked main thread at 61 cells; now ~23ms flat). The consequence: the editor never sets CodeMirror's `dark` flag, so `EDITOR_THEME` must itself supply every color the library's `&light`/`&dark` base rules would have picked (panels, tooltips, buttons, text fields = the search panel + autocomplete popup) — add a token there, not a `dark:` option. The palettes are unchanged: pygments "default" (Jupyter light) on the grey editor surface, and One Dark ported verbatim (the `@codemirror/theme-one-dark` dep is gone). `light-dark()` takes colors only, so the few non-color values that differ by scheme (syntax font weight/style) key off `<html data-color-scheme>`, an attribute `applyTheme()` in `+page.svelte` derives from that same resolved `color-scheme`. There is no `theme` prop on `LiveNotebook`/`Notebook`/`Cell`/`FileTab`. `Cell.svelte` also carries the per-cell toolbar and renders nbformat outputs. **Image outputs** (`image/png`/`jpeg`/`gif` base64, `image/svg+xml` URI-encoded) render as an `<img data-testid=output-image>` via `imageDataUrl()`; the image mime is **preferred over `text/plain`** so a matplotlib figure shows the plot, not its `<Figure … with N Axes>` repr (the kernel's inline-backend injection is what makes `image/png` present - see the Kernel bridge entry). **Double-click an image toggles fit-to-width ↔ natural pixel size** (JupyterLab behavior): in full size the `<img>` (`max-w-none`) is wrapped in an `overflow-auto` box (`data-testid=output-image-wrap`, capped `max-h-[80vh]`) so it scrolls within the output area instead of widening the notebook. The choice is **view-only** local `$state` keyed by output index (`fullImages` Set) - a re-run resets it, never touches the `.ipynb`. Because that wrapper owns its own scroll, a full-size image (like a DataFrame grid) is excluded from the outer output-scroll auto-contraction and hides that toggle (`ownsScroll = hasDataframe || hasFullImage`), so there is never a nested double-scroll. Markdown cells render to sanitized HTML (markdown-it `html:false` + DOMPurify, client-only via `browser` guard); double-click / pencil button → raw edit; Run/⌘Enter renders in place, Shift+Enter renders+advances. Type toggle is the far-right `python3`/`markdown` label. Per-notebook cell ops (add code/markdown, delete/reorder, single-kernel one-run-at-a-time; markdown "run" persists source only, no kernel) live in `src/lib/LiveNotebook.svelte`, which owns one notebook's cells + renders via `Notebook.svelte` (pure renderer). `Cell` seeds its editor doc once on mount and does NOT sync later `cell.source` changes into the editor — so `LiveNotebook` fetches authoritative cells on mount (never pre-seeds a stale snapshot). Editor source autosaves via debounced `PATCH`. Dev exposes `window.cellarViews[id]`. **Active cell** = `LiveNotebook` `activeId` (set on a Cell's `onfocusin`/`onpointerdown`, and on keyboard move); the active Cell gets a `border-primary` ring + a left accent bar (`data-active`), works in both themes. **Drag-to-reorder cells**: a per-cell grip handle (`data-testid=drag-handle`, `draggable`) + HTML5 DnD drop zones in `Notebook.svelte` → `onMoveToIndex` → the move route's `{toIndex}` → server `moveCellTo` (persists + git-clean like any move; keyboard ⌘/Ctrl+Shift+Arrow still works). **Scrollable outputs** (Jupyter "Enable Scrolling for Outputs"): a per-cell toggle contracts a tall output to a fixed-height scroll box; auto-applies above `SCROLL_THRESHOLD` px unless overridden. The choice persists in `metadata.cellar.output_scrolled` (undefined=auto / true / false) — round-trips through clean-on-save; note `cellView` in `notebook.js` now carries `metadata` so the browser sees it, set via PATCH `{scrolled}`. **Collapsible code editor** (sibling of scrollable-outputs, code cells only): the editor grows with content up to `EDITOR_MAX_PX` (576px = `max-h-[36rem]`), then a per-cell toggle (`data-testid=editor-collapse-toggle`, same style/placement as the outputs-scroll toggle) contracts it to that fixed scrollable box; auto-caps tall editors unless overridden (tri-state undefined=auto / true / false; a `ResizeObserver` on `editorEl` measures unclamped `scrollHeight`, covering content growth + the hidden→visible tab transition). Unlike `output_scrolled`, this choice is **runtime-only** like fold state — `LiveNotebook` `editorCollapsed` map persisted to `localStorage['cellar-editor-collapsed:'+absPath]`, **never** written to the `.ipynb` (zero git-diff). Markdown edit views keep the original `max-h-96` cap. **Markdown tables in text/stream output** are parsed (header + `|---|` separator) and rendered as sanitized daisyUI `table table-zebra table-xs` (same markdown-it + DOMPurify path as markdown cells); non-table text is unchanged. **Collapsible headings** (Jupyter "Collapsible Headings"): **the foldable unit is a heading *occurrence*, not a cell.** A markdown cell may hold several headings (`# Intro` / `## Setup` / `### Details` in one cell is ordinary), so `src/lib/headings.js` splits each markdown source into segments at its heading lines (fence-aware) and addresses each heading by a **fold key**: the plain cell id for a cell's *leading* heading, `<cellId>#<segIndex>` for the rest. That is what lets **every** heading level carry its own chevron (`data-testid=fold-toggle`, rendered inline beside the heading in `Cell.svelte`'s rendered markdown, not in the toolbar) - keying folds by cell id could only ever fold the first heading of a cell. A heading's section runs until the next heading of the **same or higher** level, wherever it lives: later in the same cell, or in a following cell. Nested folds compose. `computeFolding(cells, foldedIds)` returns `{hidden, segs, counts}` - whole cells to drop (`Notebook.svelte` `class:hidden`s them), per-cell segment indices an *outer* fold hides inside a still-visible cell, and the whole-cell count per fold key. A folded heading keeps its own line, tinted with an ellipsis + "N cells hidden" chip, so a collapsed section reads as collapsed. Fold state is **runtime-only** (`LiveNotebook` `foldedIds`, persisted to `localStorage['cellar-folds:'+absPath]`, restored on load) so it never touches the `.ipynb` (zero git-diff) - the deliberate contrast with `output_scrolled`, which does round-trip through `metadata.cellar`. **`LiveNotebook` owns the one fold state and the sidebar Outline is a second *view* of it, never a second copy**: `onFoldsChange(path, foldedIds, folding)` publishes it up through `+page.svelte`, `onRegisterFolds(path, toggleFold)` lets an outline chevron call straight into the notebook's own `toggleFold`. Both directions therefore stay in sync by construction. (When reporting fold state upward, assign into the map key - `notebooksFolds[path] = …`; a `{...spread}` rebuild *reads* the state the reporting `$effect` writes, and Svelte loops forever.) `headings.js` also owns `withHeadingLevel(source, level)`, the write-side inverse of `headerLevel` that the `1`-`6` shortcuts use to (re)stamp a cell's leading heading. Note the deliberate divergence: MCP's server-side section tree (`service.js`) still treats a cell's first line as its only header, since agent-facing sections are addressed by cell id.
- **MCP agent interface.** `src/lib/server/mcp/` — the agent surface (spec §4). `service.js` = transport-independent tool logic (section tree from markdown headers, tiered/summarized outputs, visibility filtering, UUID-addressed read/write/execute); `server.js` = `McpServer` + Streamable HTTP transport on its own port (`CELLAR_MCP_PORT`, default 39587 but the launcher now allocates a **free** port per instance so concurrent `cellar` runs don't collide, path `/mcp`), started once from `src/hooks.server.js`. Runs **in-process** in the SvelteKit backend so it shares the live `notebook.js` doc + `kernel.js` — restarting the kernel can't drop the MCP session/document (the #1 requirement). Per-cell hide/show via `metadata.cellar.hidden_from_agent` (allowlisted, visible by default) is honored in every map/read/search/section/result. Kernel service methods `restartKernel/interruptKernel/kernelStatus` live in `kernel.js`; richer read/write (`listCells/getCell/setVisibility/moveCellTo`) in `notebook.js`. **Coherence doctrine:** `server.js` passes an `INSTRUCTIONS` house-style string as the `McpServer` `instructions` (advisory, delivered once at connect: imports-at-top convention, check `kernel_state` first, continue the story via `get_notebook_map`, structure with markdown) + a matching `cellar_notebook_style` prompt. The read-group **`kernel_state`** tool returns the live namespace bucketed into imports/functions/classes/variables (types, shapes, dataframe columns); it reuses `inspect.js`'s shared probe (`kernelState()`), short-circuits to `{started:false}` when `kernelStatus()` is `not_started` (no force-boot) and to `{started:true,busy:true}` when the kernel is busy (never queues a probe behind a running cell), and dedupes imports by statement. Its `session_id` is the epoch the **probe itself executed in** (taken from `execute()`'s `kernel` event, never sampled around the `await`), so it always identifies the very session the returned namespace was read from; a restart between probe and reply yields `stale:true` rather than silently pairing an old epoch with a new namespace. `inspectVariables()` (sidebar) shares that probe but projects to the original `{name,type,shape,preview}` shape. **Live vs persisted execution (agent-safety invariant).** A cell's `outputs` live in the `.ipynb` and outlive the kernel *and* the server process, so they can NEVER answer "did this run in the kernel I'm talking to" — conflating the two let an agent trust a saved `run_status:"ok"` and build on variables that were never defined this session (→ `NameError`). `kernel.js` owns a monotonic **session epoch** (`kernelSession()`/`currentSessionId()`), bumped on first start, `restartKernel()`, `rebindKernel()`, and jupyter's own `autorestarting`. The epoch bumps on the **failure** paths too (a `finally` in `restartKernel()`, before the replacement start in `rebindKernel()`): once the REST restart is issued the namespace is gone even if the websocket reconnect rejects, and the epoch is monotonic + opaque, so over-invalidating is always the safe direction while a missing bump re-opens the `NameError` trap. The `autorestarting` handler is identity-guarded (`currentKernel === kernel`) so a late emission from an **outgoing** kernel cannot bump the epoch of the one that replaced it, and it re-injects `initKernel()` for parity with `restartKernel()` (an autorestart clears the namespace *and* the matplotlib backend); `rebindKernel()` disconnects the handler on teardown. Both run entry points (UI `/run` route + MCP `runCell`) stamp the epoch the run **started** in onto the runtime-only `metadata.cellar.lastRun` — captured from `execute()`'s `kernel` event and **never re-read via `currentSessionId()` on the error path**, since an autorestart mid-run bumps the epoch and would otherwise stamp the *new* one, marking a cell that never touched the fresh namespace as live (it stays null when no kernel ever started, so `ran_this_session` is false). The stamp is runtime-only in **both** directions, through one shared helper (`clean.js`'s exported `stripLastRun`, the two halves must stay in lockstep): `clean.js`'s `cleanCell` strips `lastRun` before disk (**never persist the epoch**) and `ipynb.js`'s `deserialize()` strips it on read, so a hand-edited or externally-authored `.ipynb` cannot forge `ok_session` (epochs are small monotonic ints, so a literal `session: 1` would otherwise match the first real kernel). The stamp may only ever originate from an in-process run. `service.js` derives `ran_this_session` + a split `run_status` (`ok_session`/`error_session` = executed against the live namespace; `ok_persisted`/`error_persisted` = saved output from a prior session; `unrun`; `error_kernel_unavailable`) from that stamp, never from `outputs.length` — a cell can run successfully and emit nothing (`lp = load()`). It checks the recorded status **positively** (`=== 'ok'`), so jupyter's `abort` reply degrades to `error_session` rather than a false `ok_session`. The epoch-equality rule lives in exactly one predicate (`isLiveSession`, wrapped by `ranThisSession`) - never re-derive it inline. **`error_kernel_unavailable`** covers the third case: `execute()` threw before emitting its `kernel` event (sidecar unreachable), so no session exists to stamp and the synthesized `CellarError` would otherwise fall through to `error_persisted` - telling the agent to ignore, as leftover, a failure the run it just requested raised seconds ago. Both run entry points record a `kernel_unavailable:true` marker on `lastRun` (runtime-only, so it rides the same strip-on-read/write guard); `runStatus` and `get_errors` surface it, but **only while the kernel is still unreachable** (`sid == null`) - the marker is *derived*, never trusted as sticky truth. That gate is load-bearing: the boot race (MCP serves before the sidecar finishes starting, the first `run_cell` is rejected, the sidecar comes up moments later) would otherwise leave one cell claiming a LIVE kernel-down failure forever, since only re-running *that* cell clears the flag - the same mislabel with the sign flipped. Once a kernel exists the cell falls back to `error_persisted`, which is honest. It is a status about the **run attempt** and is orthogonal to `ran_this_session`, which stays false - nothing executed in any session. A multi-cell read (`readCells`/`readSection`) samples the epoch **once** per response, so a restart landing mid-loop can't classify one payload against two sessions. **Every** read tool that surfaces saved outputs carries `ran_this_session`, `get_full_output` and `search_cells` included, so no tool hands an agent a stale output with no live/persisted signal; `get_notebook_map` carries a `kernel` header and `get_errors` flags stale errors. `execute()`'s `internal:true` opt-out keeps `inspect.js`'s probes out of `execs_this_session`. Note the deliberate scope: `ran_this_session` means "this cell executed in this epoch", **not** "the cell's current source ran" — editing after a run does not clear the stamp.

- **Follow the agent (viewport).** `LiveNotebook` tracks `agentRunningId` (set from a `run:start` whose `actor === 'agent'`, cleared on `run:end`/delete/refetch) *alongside* `runningId`, and scrolls that cell into view. `runningId` is deliberately NOT the trigger: it is set for every run (this tab's own, another tab's user run), and only an agent's run may move a viewport - a user run that yanks the page is the hostile case. Guards, in order: the tab must be `active` (a hidden pane is `display:none`, so it has no geometry - the effect re-fires on activation and follows a still-running cell); the user must not have typed inside this notebook in the last 3s (`lastTypedAt`, a capture-phase keydown listener on the notebook root; focus alone is too coarse - a cell keeps editor focus long after typing stops, and following would silently never happen); the cell must not already be visible (follow-tail: "visible" = its **top** edge is on screen, since the run affordance lives there). A cell hidden by a folded heading is revealed first (`revealCell` drops exactly the folded headers whose own section contains it) - fold state is runtime-only, so this costs a re-fold, not a git diff. The scroll runs inside `untrack()`: it reads (and `revealCell` writes) fold state, which would otherwise re-fire the effect on every edit and loop. Cell lookup is scoped to `rootEl` because **cell ids repeat across notebooks** and every open notebook stays mounted. `rootEl` is the notebook's single subtree handle, shared with the modal keyboard: it must be a real, focusable box (`tabindex=-1`), **not** `display:contents` - the shell's `overflow-y-auto` pane is the scroll ancestor, so a plain wrapper stays layout-neutral. Its scroll helper (`scrollElementIntoView`, centers or pins a cell) is deliberately distinct from the keyboard's `scrollCellIntoView(id)` (`block:'nearest'`, smallest movement): agent-follow reframes the viewport, selection must not. Selection (`activeId`) is never touched; `Cell.svelte` marks the running cell with a `warning`-hued accent bar + ring (`data-running`, `data-testid=running-bar`) that outranks the `primary` selection accent, so "executing" and "selected" read as different things.

- **Single-instance-per-folder + reap (`bin/cellar.js` step 0 + `runtime.js` + `instances.js`).** A relaunch in a folder that already has a live cellar **takes over**: it **reaps the old instance and starts fresh**, rather than leaving the old server running with stale in-memory code. This is the fix for the pile-up: old `build/index.js` servers accumulate (a launcher killed uncleanly leaves its app **reparented to init**, still listening for hours; or a still-running instance after a code update keeps serving old MCP `instructions`/tool descriptions to agents that discover its port). The loser of the lock **no longer attaches** — it **reaps the owner** (`reapInstance` on the owner's registry entry when present, so its recorded app/jupyter child pids are killed explicitly, else a synthetic entry relying on the launcher's SIGTERM cascade), then re-acquires the lock (retry loop) and boots. `waitForInstanceUrl` is now unused/removed from the launcher. `--new`/`--force` skips the whole reap+lock block to run a deliberate second instance. `cleanup()` releases the lock (pid-guarded) only if we took it, and drops our registry entry.

    - **Identity-verified reaping — the registry keys by launcher pid, but a pid is only unique while alive.** Once an instance exits (its throwaway worktree removed), the OS can hand that exact pid to an UNRELATED process; a reaper that SIGTERMs `launcherPid` on the pid alone then kills the innocent reuser (the observed bug: a fresh launch killed a different healthy cellar + its live kernel). So `reapInstance` never signals a pid unless it is **provably still that instance**: `pidReapDecision(pid, recordedStart, startedAt)` → `verifyPidIdentity` compares the pid's live `processStartTime()` to the recorded `*Start` (exact, ±`START_TIME_SKEW_MS`), falling back to the `startedAt` window for legacy entries. `true` ⇒ kill; `false` (pid reused) or `null` (unverifiable / no `ps` / Windows) ⇒ **SKIP the kill, only prune the stale entry** — killing REQUIRES a positive match, so over-pruning is the safe direction. This is why lingering stale entries are harmless. Trade-off: a **pre-registry / lost-entry** same-workspace owner carries no `*Start`, so a take-over degrades to skip-and-reclaim-the-lock (documented at the launcher call site); every current-code instance registers `launcherStart`, so the normal relaunch take-over is unaffected. **Every reap decision is logged** (candidate `launcher`/`workspace`/`reason`, the identity result, KILLED vs SKIPPED/PRUNED). The launcher/`parent-watch`/app also log their **shutdown trigger** (SIGTERM received vs. orphan self-exit vs. child-exit vs. clean) so a killed kernel is distinguishable from a crash. `parent-watch` requires `CONFIRM_STRIKES` consecutive dead readings before self-exiting. Covered by `tests/unit/instances-reap.test.ts` (spawns a real process to prove an impostor pid survives).

- **Editor edit-flush (no-run persistence).** `Cell.svelte` autosaves source on a 500ms debounce; it also `flushEdit()`s immediately on editor **blur** and window **`pagehide`**, and teardown flushes rather than cancelling - so a cell edit survives closing Cellar even inside the sub-debounce window, without a run. `LiveNotebook.editCell(id, source, {keepalive})` sets `keepalive:true` **only** from the unload flush (Cell passes `{keepalive:true}` from `pagehide`); normal page-alive autosaves stay plain `fetch` because the browser caps a keepalive body at ~64KB - routing every autosave through keepalive silently drops edits to cells >64KB (the regression to avoid). - **Kernel run queue (`src/lib/server/run-queue.js`).** One kernel serves every open notebook, so one cell runs at a time app-wide. A run requested while it is busy is **queued FIFO**, never dropped (the old behavior: the UI early-returned and MCP raced into `execute()`). The queue lives on the **server** because the kernel does - a run queued from notebook B waits behind one from notebook A, and an agent's MCP run interleaves with a user's UI run in submission order; no per-tab or per-notebook flag can see both. Entries are keyed by `(absolute notebook path, cellId)` - **cell ids repeat across documents**, so the notebook half is load-bearing. It is a *slot* primitive: it decides who runs next, never how. Both run entry points keep owning their own execution + streaming and just take a ticket first (`enqueueRun` → `await ticket.wait()` → `finally ticket.done()`); `ticket.done()` fires only **after** `run:end` is persisted + published, so the wire order stays `run:end` → next `run:start`. **Dedupe:** a cell already running or already queued is not enqueued twice; the second submission instead **refreshes the pending entry's source** (`ticket.source()` is read at dequeue), which is what makes dropping it safe - a queued cell runs what you last submitted. **Cancellation:** `clearRunQueue()` drops every *pending* run and is called from `restartKernel`/`interruptKernel`/`rebindKernel`/jupyter's `autorestarting` (in `kernel.js`, **before** the restart is issued) - queued work was submitted against a namespace that is about to vanish, and running it against a fresh kernel would execute cell N+1 without cell N's definitions. The *active* run is not touched: the kernel op ends it, and its owner then releases into an empty queue. `deleteCell` calls `cancelRun` so a removed cell can't dequeue later; the `/run` route's stream `cancel()` drops its own pending entry (identity-checked via `ticket.cancel()`, **never by key**, so a duplicate request's stream closing cannot cancel the real pending run). A cancelled entry rejects with `RunCancelled`; `enqueueRun` pre-attaches a no-op `.catch` so an owner that already returned never surfaces an unhandled rejection.
- **Queue state is a broadcast snapshot, not a delta.** Every queue change (and every run start, including the uncontended one) publishes `queue:changed` via `events.js`'s **`publishGlobal`** - it carries `{running, queue:[{nb,cellId,actor,position}]}` for the WHOLE kernel, has **no `nb` and no `seq`** (it is a full snapshot, so a missed one self-heals), and is dispatched in `LiveNotebook` *before* the per-notebook `nb`/`seq` filter and *without* `originId` echo-suppression - a queue is shared state, not one tab's action. The SSE route seeds one on connect and `events-client.js` replays the latest to a late subscriber, so a tab connecting (or a notebook mounting) mid-run sees both the queue **and** the running cell; the snapshot's `running` is why `LiveNotebook` reconciles `runningId` from it (a tab that missed `run:start` would otherwise render cells "queued · 1" behind a cell it shows as idle). `run:start` still solely owns `agentRunningId` - adopting a running cell must not scroll a tab that never saw the agent start it. The `/run` NDJSON gains `run:start` (the initiating tab drops its own SSE echo, so this is how it learns its run actually *started* rather than merely being accepted), plus terminal `run:duplicate` / `run:cancelled` frames that close the stream without ever touching the cell's outputs. `runCell`'s `finally` clears `runningId` only when it truly `started`, so a refused duplicate can't erase a live indicator. **`runBusy` in `+page.svelte` is no longer a gate** - it is an in-flight *counter* (`runsInFlight`) feeding the kernel badge; the `busy` prop is gone from `LiveNotebook`.
- **Queued-cell affordance.** `Cell.svelte` `queuedPosition` (null | 1-based **global** position) → a static `bg-warning/50` left bar + `border-warning/30` + an amber clock and `queued · N` label, the deliberately quieter sibling of the running affordance (`bg-warning` pulsing bar + spinner), which outranks it (`isQueued = queuedPosition != null && !running`). Label text is `base-content`, **not** `warning/70`: a dimmed warning washes out to near-illegible on a light theme. Positions are global on purpose - a cell queued here may be waiting behind another notebook's cell.
- **MCP run tools + the queue.** `run_cell` **waits** its turn and then returns the real outputs annotated `queued:true` / `queue_position` / `waited_ms` - it does *not* return early as "queued". Waiting is what keeps `run_cells`/`run_all`/`run_range` meaning "these ran, here is what they printed"; they drive `runCell` sequentially and depend on its outputs. The one early return is the **dedupe** case: `{status:'queued', queue_position}` if that cell is already pending, `{status:'running'}` if it is executing right now - so an agent is told it was accepted and never that a run finished when it did not. A dropped pending run returns `{status:'cancelled', reason}` and `runCells` **stops** there (the rest would run against a namespace their predecessors never populated). `runCell` pins `nb = getActiveNotebookPath()` at call time and threads it through `getCell`/`setOutputs`, since the UI may focus another notebook while the run waits. New read tool **`run_queue`**; `INSTRUCTIONS` gained a "one kernel, one run at a time" clause (don't interrupt/restart just because a run is slow - that drops the user's queued runs too). **Long-run keepalive (the silent-hang fix).** Because a run tool `await`s the whole run and holds the HTTP response open, a cell longer than the MCP CLIENT's per-tool-call timeout (Claude Code default ~60s) made the client give up and DISCARD the eventual result - the agent hung and never got the output even though the cell finished server-side. Fix lives entirely in `mcp/server.ts` `withProgress(extra, fn)`: it wraps every run tool handler (`run_cell`/`add_and_run`/`run_cells`/`run_all`/`run_stale`/`run_range`) and, while `fn` is pending, emits an MCP **progress notification** every `PROGRESS_INTERVAL_MS` (15s) against the request's `_meta.progressToken` via `extra.sendNotification` - which resets the client's timeout (when it sets `resetTimeoutOnProgress`, as Claude Code does) so the FINAL result is delivered when the run completes. It's a **wall-clock heartbeat, NOT output-driven** (the canonical `time.sleep(90)` emits nothing until it ends); a client that sent no progressToken gets a no-op passthrough (never push unsolicited notifications); send errors are swallowed and the timer is `unref`'d. The tool contract is unchanged (still returns the full outputs at the end). The `httpServer.requestTimeout = 0` / `timeout = 0` in `startMcpServer` is belt-and-suspenders (Node's 300s `requestTimeout` measures request RECEIPT, not the long response, so it never actually cut a run). The stdio↔HTTP bridge relays notifications verbatim and imposes no per-request timeout. **Residual cap:** if a client sets a hard `maxTotalTimeout`, no amount of progress overrides it - that's a client config choice, not something Cellar can fix. Covered by `tests/unit/mcp-run-progress.test.ts` (fake-clock heartbeat + result delivery).

Never shell out to the `databricks` CLI, and **never** glob for the VS Code extension's bundled binary (the boilerplate this replaced did; it is the thing we deliberately do not do). **The split is the design:** *listing* (profiles/clusters/catalogs/schemas/tables) runs **server-side** in a short-lived subprocess of the project venv's python (`projectPython()` = `CELLAR_PROJECT_VENV`, the same interpreter the kernel uses - so "can the kernel import the SDK" and "can our listing import the SDK" can never disagree); *the session* runs **in the kernel**, because `spark` must live where the user's cells can reach it. One python script (`PROBE`) serves every listing op, driven by a JSON command in `argv[1]`; it **always prints exactly one `__CELLAR_DBX__`-prefixed JSON line and never raises**, so a missing package / bad profile / unreachable workspace arrives as `{ok:false, code, message}` rather than a traceback we would have to guess at. `code` is the contract: the sidebar's `REMEDY` map turns it into actionable copy, and `statusFor()` into an HTTP status. Embedded as a JS string (not a sibling `.py`) because `$lib/server` is bundled into `build/` - the same reason `inspect.js` embeds its probe. Both halves scrub `DATABRICKS_*`/`SPARK_CONNECT_*` from the environment, **keeping `DATABRICKS_CONFIG_FILE`** (the one deliberate divergence from the boilerplate: it is what says where the config we just parsed lives). Profiles are parsed **in Node** so the picker renders before the SDK is installed; a section without a `host` is not a profile (the CLI writes `[__settings__]` into that same file). **Connection state is epoch-scoped**: `connect()` stamps the kernel-session epoch from `execute()`'s `kernel` event, and `connectionStatus()` reconciles against `currentSessionId()` on every read - the exact `lastRun` doctrine, for the exact reason (a restart destroys `spark`; reporting "connected" from a dead namespace hands the user a `NameError`). Connect/disconnect/preview run through `execute(..., {internal:true})`, so they bypass the run queue (like `inspect.js`'s probes) and never inflate `execs_this_session`; jupyter serializes them behind a running cell. A connect can legitimately take minutes (Databricks Connect starts a cold cluster), which is why the route just waits. `databricks-sdk`/`databricks-connect` install into the **project** venv via `venv.js`'s `installPackages` (the one place Cellar adds anything to a project venv); `databricks-connect` must match the cluster's DBR major.minor, hence the version pin. **Agent side:** `kernel_state` + `get_notebook_map` carry a `databricks` block, `INSTRUCTIONS` clause 9 tells the agent to use `spark`/`w` instead of writing boilerplate, and five read-only MCP tools (`databricks_status`, `databricks_list_catalogs/_schemas/_tables`, `databricks_preview_table`) reuse the **same** server-side plumbing as the UI tree (`forAgent`), so the two views cannot disagree. **Connecting is a human action** - every agent tool fails `not_connected` and says to ask the user. The UI's table preview appends+runs a real cell (`spark.read.table(...).limit(N).toPandas()`) because there the cell *is* the deliverable; the agent's `preview_table` reads through `spark` and touches nothing, because an agent peeking at a table must not append to the human's document. `LiveNotebook`'s `onRegisterApi` (`insertAndRunCode`) is how the sidebar reaches the active notebook - same registry shape as `onRegisterFolds`; it deliberately does **not** move `activeId` (focus must follow selection, and the user's focus is in the sidebar).
- **User-choosable imports cell (unpinned).** ONE code cell per notebook, `metadata.cellar.role = 'imports'` (rides the `cellar` allowlist, so it round-trips clean-on-save with zero git noise), living at **any** index. The user designates any Python code cell via the per-cell **⋮ actions menu** (`Cell.svelte`, `data-testid=cell-actions` → `toggle-imports-role`): "Mark as imports cell" / "Unmark as imports cell", wired `onSetRole` → `LiveNotebook.setRole` → `PATCH /api/cells/[id] {role}` → `setCellRole` (which enforces one-per-notebook: designating a cell strips the role from any other, atomically + optimistically) → `cell:role` SSE. A pin-glyph **"imports" badge** (`data-testid=imports-badge`) is the position-independent marker. Three things fill the cell: that manual designation, the **Consolidate imports** action (navbar app menu → `POST /api/notebooks/imports` → `consolidateImports`) and **agent routing** in the MCP write tools. A human typing `import os` into cell 7 is *never* touched — that is the deliberate line. `src/lib/importsRole.ts` is the shared browser-safe identity (`isImportsCell`, `importsCellIndex`, position-independent); `clampMoveIndex` is now the **identity** function (the top-pin was removed) — kept at every move site (`moveCell`/`moveCellTo` server, `Notebook.svelte` client) as the one seam to reintroduce a positional rule if ever wanted. `ensureImportsCell` no longer hoists a designated cell; it returns it in place, and only creates/adopts a top cell as the fallback when NONE is designated. `setCellType(…,'markdown'|'sql')` demotes the role (only a Python code cell can hold it, else it would strand every future routed import).
  **Parsing is a JS tokenizer, not the kernel's `ast` and not a regex** (`src/lib/server/imports.js`, pure): the kernel would force-boot / queue behind a running cell and make a structural edit depend on the runtime, while a regex cannot see that `import evil` inside a triple-quoted string, `if TYPE_CHECKING: import pandas`, or `def f(): import json` are not module-level. `logicalLines()` understands comments, all four string forms + prefixes + escapes, bracket and backslash continuation, and reports each logical line's **indent** — only indent-0 lines whose first token is `import`/`from` are lifted. Anything that does not parse into a canonically re-renderable record is **left exactly where it is**. `mergeImportSources` returns the existing source **untouched** when nothing is new, which is what makes consolidate idempotent in the strong sense (second run: zero document change, zero kernel re-run). Ordering: `__future__`, plain `import`, absolute `from`, relative `from` — sorted by code unit, deliberately NOT isort's stdlib/third-party grouping (needs a package classification Cellar has no business guessing).
  **Adoption:** an existing role cell wins; else a first cell that is *only* imports/comments/blanks is adopted (`isImportsOnly`); else a fresh cell is inserted at 0 (`addCellAt`, which is why `cell:added` now carries an explicit `index` — `afterId` cannot express "at the very top"). New event `cell:role`. Consolidate deletes cells its sweep emptied (never the last cell, never the imports cell) and runs the imports cell **only when something moved**.
  **Agent routing** (`service.js` `routeOne` → `finishImportRouting`): `add_cell`/`add_cells`/`edit_cell`/`add_and_run` lift module-level imports out of the submitted code into the imports cell, then run it **once** (all specs of one `add_cells` merge into one cell → one run, not one run per cell); `add_and_run` routes *before* creating the cell, so the code that needs `pd` finds it. Opt-out per call: **`route_imports: false`**. Editing the imports cell itself never routes into itself. Source that is *only* imports creates no cell (`routed_to_imports: true`). The `run_*` tools are deliberately **NOT** routed: they submit no code, and yanking imports out of an existing cell at run time would rewrite a cell a human may have authored (and `run_all` would strip the whole notebook).
- **nbdev-style cell export to a `.py` module (`src/lib/server/export-py.ts` + `src/lib/exportRole.ts`).** A per-code-cell **on/off** flag `metadata.cellar.export = true` (nbdev's `#|export`, rides the `cellar` allowlist → round-trips clean-on-save with zero git noise) plus a **notebook-level** target `notebook.metadata.cellar.export_target` (a workspace-relative `.py` path, nbdev's `#|default_exp`; also honored as a `#|default_exp <module>` directive in any cell, dotted→path). `export-py.ts` reads the marked cells IN ORDER and writes a clean module: header `# AUTOGENERATED BY CELLAR! DO NOT EDIT!` + `# Source notebook:` line, a best-effort `__all__` (top-level `def`/`class`/simple assignments via `imports.ts`'s `logicalLines`, private `_names` skipped), then the sources joined by blank lines + trailing `\n` - **deterministic + idempotent** (skips the write when on-disk bytes match, so re-export is a zero git-diff no-op). This is DISTINCT from the jupytext whole-notebook `.py` mirror (`jupytext-actions.ts`): that mirrors every cell with markers; this is a lean library module of only the marked cells, **committed to git** (NOT gitignored, per nbdev). **Two triggers:** (1) **auto-on-save** - `notebook.ts` `persist()` calls `autoExportPy(doc)` after every write (best-effort try/catch so a bad target never breaks the save; no-op unless a target is set AND ≥1 cell marked; skipped for `.py` jpFormat docs which carry no cell metadata); (2) **manual** - `exportPy()` surfaces real errors, reached via the notebook's top **export bar** (`Notebook.svelte`, `data-testid=export-bar`: target input + "N cells marked" + "Export to .py" button + inline feedback, shown once a target is set OR any cell is marked) and the navbar app-menu "Export to .py" item (`+page.svelte` → `activeNotebookApi().exportPy()` → toast). **Wiring:** per-cell toggle in the ⋮ actions menu (`Cell.svelte`, `data-testid=toggle-export`) + a green **"export" badge** (`data-testid=export-badge`) → `onSetExport` → `LiveNotebook.setExport` → `PATCH /api/cells/[id] {export}` → `setCellExport` → `cell:export` SSE; target via `POST /api/notebooks/export-py {op:'set-target'|'export'}` → `setExportTarget`/`exportPy` → `notebook:export-target` SSE. `getNotebook`/`NotebookView` carry `exportTarget` so the browser seeds it on load. `setCellType(…,'markdown'|'sql')` drops the export flag (only a Python code cell can hold it). Known limit: `__all__` misses dynamically-created names + augmented assignments (documented in `export-py.ts`). Pure logic covered by `tests/unit/export-py.test.ts`.
- **Cell staleness + dependency graph.** A code cell that ran this session but whose inputs changed since is **stale** - the flagship correctness signal for an agent-first notebook (cells run out of order). Three layers: (1) **dataflow** `src/lib/server/dataflow.js` runs a Python **`ast` + `symtable`** analysis in a project-venv subprocess (`projectPython()`, like `databricks.js` - NOT the kernel and NOT a regex, for the same reasons `imports.js` avoids both: no force-boot, no queue, works with no kernel; results cached by source string) to get each cell's `{defines, uses}`. **The split is the design, do not collapse it.** *Module scope* is an ordered **`ast` load-before-store walk**: a `Load` of a name the cell has not yet bound is a use, one it already bound is not. Order is the whole question and `symtable` cannot express it - it knows only that a name is referenced *somewhere* and assigned *somewhere*, so it cannot tell `x = x + 10` (reads upstream) from `df = read(); df.head()` (reads its own). Both rules available to it are wrong: subtracting assigned names (`uses = referenced − defined`, the rule this replaced) dropped the `x = x + 10` read and left the cell falsely `fresh`, while counting every reference over-reports the `df = read(); df.head()` idiom into a **permanent, uncleanable** stale chip (`upstream_unrun` keys off the upstream's run state, and `run_stale` only re-runs cells already stale). The walk drives on **evaluation order, not field order** (`Assign._fields` is (targets, value) but the RHS evaluates first; `For` evaluates `iter` before binding `target`), which is also what reaches `count += 1` - `symtable` reports `is_referenced()==False` for an `AugAssign` target. *Nested scopes* stay **`symtable`'s**, order-INSENSITIVE (`used_nested - defines`): a def/lambda body is DEFERRED, so "bound so far" for it is "bound anywhere in the cell", which is what keeps `def f(): return pd.x` a use of `pd` while a recursive `def fact(): fact(n-1)` gets no self-edge. Keeping `symtable` there is deliberate - global vs. local vs. free vs. `nonlocal` is exactly what it already gets right. The ast walk therefore never descends into a def/lambda body, only into the parts that DO evaluate at module scope (decorators, defaults, annotations, bases). Covered by `tests/unit/dataflow-load-before-store.test.ts`, which spawns the REAL probe - its positives and negatives fail in opposite directions and are one contract; (2) the **pure rule** `src/lib/staleness.js` (`computeStaleness(cells, dataflow, sid)`) builds the preceding-definer graph (B depends on A when B uses a name A defines; nearest preceding definer wins; acyclic, so doc order = topological order) and marks each code cell `not_run`/`fresh`/`stale`/`n/a`. A cell is stale if it was edited after it last ran (`metadata.cellar.editedAt`, a NEW runtime-only stamp set in `setSource`, stripped from disk by `clean.js`'s renamed `stripRuntimeMeta` alongside `lastRun`), or a (transitive) upstream was edited / re-ran / didn't run this session, all vs the existing `lastRun.{at,session}`. Never-run reads `not_run`, distinct from stale. (3) **Surfaces**: server helper `getNotebookStaleness(nb)` (in dataflow.js) is shared by `GET /api/notebooks/staleness?path=` (the UI fetches it, debounced, on load/run-end/edit/structural change) and the MCP layer. UI: `LiveNotebook` fetches → `Notebook` → `Cell` renders an amber "stale" chip (own `--cellar-stale` token in app.css, distinct from the `warning`-hued running/queued affordances) + a quieter "not run" hint; navbar app-menu adds Run stale / Run cells above / Run cells below (delegated to the active notebook via the existing `onRegisterApi` registry - `runAbove`/`runBelow`/`runStale`, run sequentially in doc/dependency order). MCP: `get_notebook_map` carries `stale_state`/`stale`/`stale_reason`/`stale_upstream` per cell, `kernel_state` carries `stale_cells:[{id,reason,upstream}]`, `read_cell`/`read_cells`/`read_by_location`/`read_section`/`run_cell` carry the same (those read tools became **async** to compute it), new **`run_stale`** tool re-runs everything stale in dependency order, and `INSTRUCTIONS` clause 3 gained a staleness doctrine (re-run/distrust stale cells; editing a cell stales its dependents). Staleness is computed fresh each time, never persisted (zero git diff). **Staleness is static analysis + timestamps, NOT a runtime check** - `lastRun` carries `{at,durationMs,actor,status,session}`, never the names a cell read or wrote, and nothing inspects the namespace. So it is a **floor**: every blind spot in the graph is a blind spot in the verdict, and there is **no backstop** - `find_symbol`/`cell_impact` are derived from the SAME graph (`symbolGraph.ts`) and under-report identically, so never document one as catching what another missed (the false claim removed in `1812cea`). The one runtime-derived input is `lastRun.session`: a restart bumps the epoch and correctly resets everything to `not_run` - epoch tracking, not name tracking. Known limits (documented in `staleness.ts`): dynamic names (`exec`, `globals()`), forward references, a **conditional bind** (`if flag: df = load()` then `df.head()` - the sequential walk counts it bound, so no upstream edge; no worse than what shipped before), and an **`AugAssign` to a `global`-declared name inside a function** (`def g(): global c; c += 1` - a deferred body, so it is `symtable`'s half and hits its `is_referenced` blind spot; module-scope `count += 1` IS caught). **Read-then-rebind is FIXED** (the `ast` load-before-store walk) - do not re-document it as a limit, and if you are tempted to "simplify" the ast walk back into a `symtable` subtraction, read the `dataflow.ts` header first: that subtraction is the bug.
- **SQL cells: the `-- >> name` result binding (`src/lib/server/sql.ts`).** A SQL cell (`metadata.cellar.language='sql'`, see `cellLanguage.ts`) is compiled to Python at RUN time only (`run.ts`: `sqlToPython`) and executed by the ordinary Python kernel against the live `spark`; the stored source stays SQL. **The binding syntax:** a cell may open with `-- >> sales_df` to bind its result to `sales_df`. It is a plain SQL **line comment** on purpose - the cell still parses as SQL in any other tool, survives a round-trip through plain Jupyter, the CodeMirror `sql()` grammar highlights it as a comment, and an agent reads it over MCP with no new tool. Recognized **only as the first non-blank line** (never scanned for in the body, so a `-- >>` mid-query stays an ordinary comment) and **stripped** before the query reaches `spark.sql`. **The binding rule:** a named cell binds **BOTH** `<name>` **and** `_sql_df` - `_sql_df` is documented to agents (`mcp/server.ts` clause 10) as the "last SQL result" chain-off point and that contract must keep holding, so the named var is the *stable* binding and `_sql_df` stays the (last-write-wins) alias it always was. **No prefix line ⇒ byte-for-byte the legacy output** (one assignment to `_sql_df`, no alias line) - existing notebooks are unmoved; the display expression is always `<var>.limit(SQL_ROW_CAP).toPandas()` on the LAZY Spark DF, never the capped pandas one. The name is interpolated into generated Python, so it is **validated, never merely quoted** (strict ASCII identifier + not a Python keyword - deliberately stricter than Python's own unicode rule - **and not a name Cellar itself binds in the kernel**: `spark`/`w` are legal identifiers, so without that reserved set `-- >> spark` would compile to `spark = spark.sql(...)` and silently destroy the live session the next cell needs); an unusable name compiles to a lone `raise RuntimeError(<why>)` so the cell **fails visibly** instead of emitting broken Python or silently ignoring the line. **`parseSqlCell` is the single source of truth for both halves** and must stay so: `sqlToPython` compiles from it, and `dataflow.ts` reads its `resultVars` to give each SQL cell a **synthetic `defines`** - SQL cells stay OUT of the probe (their source is SQL; `ast`/`symtable` would misparse it), but their run really does bind names, so without the synthetic contribution a Python cell reading a SQL result gets no upstream edge and never goes stale when the query is edited. A SQL cell is a graph **source**: `uses` is always `[]` (reading table names out of SQL is lineage analysis, deliberately out of scope). Covered by `tests/unit/sql-binding.test.ts`.
- **Every `nb` argument is resolved to an ABSOLUTE path, at every layer.** `kernel.ts`'s `kernels` Map (like `notebook.ts`'s `docs` Map) is keyed by absolute path, so a workspace-relative `nb` (`notebook.ipynb` - how the BROWSER addresses everything) must go through `resolveNotebookPath` or it silently misses every entry. `kernel.ts`'s `resolveNb` used to pass its argument through verbatim, against its own docstring: `currentSessionId('notebook.ipynb')` returned null, so `getNotebookStaleness` reconciled against "no kernel session" and reported **every cell `not_run`** - the flagship staleness signal read dead in the UI for all cells while MCP (which addresses notebooks absolutely) worked fine, so nothing pointed at the cause. A relative path also became its own Map key, i.e. a second kernel for one notebook. Pinned by `tests/unit/staleness-relative-path.test.ts`; when adding a `kernel.ts` entry point taking `nb`, route it through `resolveNb` and never key off the raw argument.
- **`src/lib/server/run.js` = the one cell-execution core.** The UI `/run` route, MCP `runCell`, and `runImportsCell` all call `executeCellRun({nb, cellId, actor, source, originId, onEvent})`: execute → persist outputs → stamp the runtime-only `lastRun` (session epoch captured from `execute()`'s `kernel` event, never re-read) → publish `run:start`/`run:output`/`run:end`. It is NOT the queue: claiming the kernel stays with each caller, because each answers a refused (`duplicate`) or dropped (`cancelled`) ticket in its own shape. `onEvent` is what the UI route forwards down its NDJSON stream. A third run path can no longer drift from the run_status doctrine.
- **Notebook checkpoints (snapshot / undo).** `src/lib/server/checkpoints.js` snapshots a notebook's full cells (source + outputs + metadata, deep-cloned) to a restorable checkpoint. **Storage** mirrors `ui-state.js`: a single `.cellar/checkpoints.json` (gitignored, per-project, port-independent), keyed by workspace-relative path, bounded to 25/notebook (FIFO); outputs included but a snapshot over 2 MB drops them (`outputsTruncated`). - **ipywidgets support (progress bars + common INTERACTIVE widgets, two-way).** tqdm bars AND interactive controls (`IntSlider`/`FloatSlider`/range, `IntText`/`Bounded*`, `Checkbox`/`ToggleButton`/`Valid`, `Dropdown`/`RadioButtons`/`Select`/`ToggleButtons`, `Text`/`Textarea`/`Password`, `Button`, `Box`/`HBox`/`VBox`/`GridBox`, `Output`) render live, and interactive ones send changes BACK to the kernel. **Receive (kernel→FE, #86):** `kernel.ts` `registerWidgetComm` (once per connection - `restart()`/autorestart keep the connection+target, a rebind rebuilds both) does `registerCommTarget('jupyter.widget', …)` for every `comm_open`+`comm_msg` `update`. **Registering this target also SILENCES @jupyterlab's `Exception opening new comm` spam** - thrown for ANY `jupyter.widget` comm_open with no registered target - so the callback must handle ALL widget types, never throw. Server store `src/lib/server/widgets.ts` keeps `comm_id → {state}` (cleared by `beginSession()`), rebroadcasting `widget:open`/`update`/`close`/`clear` as **global** SSE (`publishGlobal` - model ids unique per session, one store spans all notebooks); the events route seeds `widget:sync` on connect. **Send (FE→kernel, the interactive core):** `kernel.ts` keeps the live `IComm`s in `widgetComms` (cleared in `beginSession`) and `sendWidgetComm(commId, data)` calls `comm.send`; route `POST /api/widgets/[commId]` forwards `{method:'update',state:{<trait>:<value>}}` (a value/`index` change) or `{method:'custom',content:{event:'click'}}` (a Button), so ipywidgets updates the Python model + fires `observe`/`interact` callbacks. The route ALSO `updateWidget`s the store (optimistic broadcast) because the kernel does NOT echo a frontend-originated change back as a plain `update` - only ipywidgets-8 `echo_update`, which the receive path **deliberately ignores** (applying it mid-drag snaps the slider back). **Client** `src/lib/widgetActions.ts` posts on interaction: continuous controls (slider drag) THROTTLE (leading+trailing, `THROTTLE_MS=80`), discrete ones flush immediately; `widgetStore.svelte.ts` `setWidgetTrait` optimistically updates locally for lag-free feedback, then reconciles via SSE. A reopened notebook shows no live widget (state is volatile, by design). Covered by `tests/unit/clean-on-save.test.ts`, `widget-model.test.ts`, `widget-output-capture.test.ts`.
- **jupytext / Databricks `.py` notebooks (`src/lib/server/jupytext.js` + `jupytext-actions.js`).** A `.py` opens as a **live, kernel-attached** notebook exactly like an `.ipynb` — the `.py` ⇄ cells boundary is `jupytext.js`, called from `notebook.js` `loadDoc`/`persist` at the same seam `ipynb.js` sits at, so everything above it (cell ops, run, events, MCP, live-sync) is **format-agnostic**. - **Export to self-contained HTML (`src/lib/server/export-html.js` + `GET /api/notebooks/export`).** Navbar app menu → "Export to HTML" (`onExportHtml` in `+page.svelte` → a hidden `<a download>` hitting the route with `?path=`) downloads ONE portable `.html` of the active notebook. **Server-side re-render of the notebook MODEL, not a DOM serialization:** the route reads `listCells(path)` (persisted outputs straight from the `.ipynb` — it NEVER touches the kernel, so export reflects the last run) and `renderNotebookHtml` emits a full standalone document with all CSS inlined and images as data URIs. It does NOT invent renderings Cellar lacks (a DataFrame exports as its text repr, not `text/html`). Python code is highlighted by a compact self-contained tokenizer in the module (CodeMirror is browser-only; a new dep wasn't worth it) that fails safe to escaped plain text. Default output is `<name>.html` alongside the notebook; an explicit `path` is workspace-relative (a `.html` extension is appended if missing). **Both paths go through `resolveInWorkspace`**, so a traversal/absolute-escape is refused - nothing is written outside the workspace. `hide_code` maps to the same report path as `?hideCode` (omit = follow the saved setting). Covered by `tests/unit/mcp-export-html.test.ts`.
- **Hide code input (report view) - per-cell + notebook-wide.** Hide a code cell's INPUT (editor) while keeping its OUTPUT visible - a clean output-only report view, display-only (source untouched, cell still runs, `.ipynb` git-clean). **Distinct from `editorCollapsed`** (the tall-editor scroll-box contraction, runtime-only): hide-code is an EXPLICIT, PERSISTED hide regardless of length. **Two flags in the allowlisted `cellar` namespace (round-trip through clean-on-save):** per-cell `metadata.cellar.hide_input` (tri-state: undefined = follow the notebook default, true = hidden, false = shown) and notebook-level `notebook.metadata.cellar.hide_all_code` (report view). **Precedence:** `effectiveHidden = hide_input ?? hide_all_code` - a per-cell explicit choice always wins; the notebook-wide toggle is the default for cells with no explicit choice, so a per-cell "show code" reveal works even under global hide. **Server** (`notebook.ts`): `setHideInput` (code cells only; emits `cell:hide-input`) + `setHideAllCode` (emits `notebook:hide-all-code`), mirroring `setOutputScrolled`/`setHeaderNumbering`; `getNotebook` carries `hideAllCode`. **API:** `PATCH /api/cells/[id] {hideInput}`; `POST /api/notebooks/hide-code {hidden}`. **Client:** `LiveNotebook` `setHideInput`/`toggleHideAllCode` (optimistic + persisted), `hideAllCode` state (loaded + kept live via SSE + published up via `onHideAllCodeChange`); both events flow through `applyStructuralEvent`. `Cell.svelte` derives `codeHidden`, hides the editor block (same `hidden` mechanism as markdown rendered-mode, so CodeMirror stays mounted) and shows a slim, discoverable **"show code" bar** (`data-testid=show-code`) that reveals in place; the ⋮ actions menu (now shown for ALL code cells incl. SQL, not just Python) gains a Hide/Show code item (`data-testid=toggle-hide-input`). The toolbar (run controls, run-status, running/queued affordances) stays visible when hidden, so a running hidden cell is legible and a run still updates the visible output. **Notebook-wide toggle** lives in the Navbar app-menu "View" section (`data-testid=toggle-hide-all-code`), state reflected from the active notebook. Markdown cells are unaffected (no code to hide). **HTML export honors hide-code (report-style export).** `renderNotebookHtml({cells, hideAllCode})` (`export-html.ts`) mirrors `Cell.svelte`'s `codeHidden` rule EXACTLY (`codeIsHidden` = per-cell `hide_input ?? hideAllCode`, markdown never hides): a hidden code cell renders **output-only**, and a hidden code cell with NO output (imports, `df = load()`) is dropped from the report entirely, so a report-view export reads as pure markdown + results. The route `GET /api/notebooks/export` couples to the saved notebook setting by default (`getHideAllCode(path)`) so a notebook read as a report in Cellar exports as one; `?hideCode=1|0` forces report style on/off for a one-off export regardless. Covered by `tests/unit/export-html.test.ts`.
- **Notebook-level display settings + their agent surface (header numbering, report view).** Both live in the allowlisted `cellar` namespace on the NOTEBOOK's metadata (`header_numbering` = which heading levels 1-6 auto-number, `hide_all_code` = report view), both are **display-only**, and both are readable AND settable by the agent. **Numbering is computed at render time and NEVER written into a cell's source** (`computeHeadingNumbers` in `headings.ts`, pure + hierarchical over the enabled levels; `setHeaderNumbering`/`setHideAllCode` in `notebook.ts` persist only the setting and emit `notebook:header-numbering`/`notebook:hide-all-code`). **MCP:** `set_header_numbering` / `set_report_view` tools (`server.ts`), thin session-targeted wrappers in `mcp/service.ts`; `get_notebook_map` carries a **`display`** block (`{header_numbering, report_view}`) and each numbered section its rendered `number`, and `read_section`'s header carries its `number`. `read_cell`/`read_cells`/`search_cells` deliberately do NOT - they return raw source, which is exactly where the number is not. The whole point is `INSTRUCTIONS` clause 5: agents hand-number headers (`## 1. Setup`) because nothing told them Cellar already does it. **The double-numbering hazard is by design, do NOT "fix" it:** a source `## 1. Setup` under numbering renders `1. 1. ## Application shell (navbar + sidebar)

Wraps the notebook in a JupyterLab-style shell. `src/routes/+page.svelte` is the shell: it owns tabs/sidebar/kernel-controls/theme and hosts one `LiveNotebook` per open notebook tab (default + each opened `.ipynb`), all kept mounted+hidden so editor/run state survives tab switches. It no longer owns `cells`: each `LiveNotebook` owns its own cells and reports the live array up via `onCellsChange(path, cells)` into a path-keyed `notebooksCells` map; the sidebar (outline/search/variables) reads `activeCells` for the focused notebook (falls back to the default when a plain file tab is active). A single shared kernel → runs are serialized across notebooks by the **server-side FIFO run queue** (see the Kernel run queue entry above); the shell's `runBusy` is only a derived in-flight counter for the kernel badge, no longer a gate.

- **Navbar** (`src/lib/Navbar.svelte`): app-menu dropdown → `src/lib/Settings.svelte` modal (theme toggle `dim`↔`nord`, persisted to `localStorage['cellar-theme']`, applied via `<html data-theme>`); editor-style **wrapping** tab bar (tabs flow onto more rows via `flex-wrap`, never clip). Every tab (including the notebook) is closable and part of session memory; tabs carry file-type icons + a dirty dot; `preview` tabs render italic.
- **File tabs** (`src/lib/FileTab.svelte`): CodeMirror editor over a workspace file, ⌘S save; syntax-highlighted by file extension (`langFor`: python / markdown / json+ipynb / yaml / toml, else plain text), sharing `editorTheme.js`'s one static `EDITOR_THEME` with cells. TOML has no lezer grammar, so it uses the official `@codemirror/legacy-modes` stream mode via `StreamLanguage`; its tokens land on the same highlight tags as every other language, so both themes style it with no theme-side work (`.toml`/`.yaml`/`.yml` share the `config` icon in `fileIcons.js`). Dirty state bubbles up to the tab bar. Notebook + all file tabs stay mounted (hidden via `.hidden`) so editor/run state survives tab switches.
- **Sidebar** (`src/lib/Sidebar.svelte`, foldable sections; whole sidebar collapsible + **drag-resizable** via the `sidebar-resizer` handle in `+page.svelte`, width persisted to `localStorage['cellar-sidebar-width']`, bounds 180–560px): **Files** (`src/lib/FileTreeNode.svelte`, recursive), **Kernels** (`kernelsSection` in `Sidebar.svelte`: **one card per kernel** — Cellar runs one kernel PER notebook (Phase 3), so the section renders `{#each kernelCards}` (a `KernelCard[]` derived in `+page.svelte`) with the **live kernel count** in the header. The card set = every LIVE kernel (from `listKernels()` via `/api/kernel` + the `kernel:status` SSE snapshot, so a closed-tab-but-alive kernel still shows) **plus the ACTIVE notebook** even with no kernel (a just-opened notebook reads "not started"). So shutting down a **non-active** notebook's kernel drops its card, while the notebook you're looking at always has one. Each card carries its own status badge + **Interrupt / Restart (=wipe namespace) / Shut down (=free the process, drop the card)**, all disabled until that notebook has a live kernel (`card.hasKernel`); the card's name focuses its tab (`onFocusNotebook`) and the active one is dotted. Live updates arrive via the `kernel:status` global SSE snapshot (published from `kernel.ts` on start / busy-idle flip / restart / shutdown / rebind, seeded on connect + replayed to late subscribers like `queue:changed`), so two notebooks running show two busy cards with no reload. **The broadcast is deduped + has a coalescing (debounced) fast path, and a busy/idle flip fans out ONLY while a USER (non-internal) run is in flight** (`NotebookKernel.userRuns`, gated in `kernel.ts`'s `statusHandler`; `scheduleKernelStatus` coalesces a run's flip burst, `emitKernelStatus` skips a byte-identical snapshot). A flip caused solely by an internal probe (inspect/variables) or a startup injection leaves `userRuns` at 0 and is suppressed - so those never flicker a card nor fan a redundant snapshot to every tab. Do NOT restore an unconditional `publishKernelStatus()` on every flip; user runs still broadcast busy→idle because `execute()` increments `userRuns` (and schedules a broadcast at the run boundary). Seeding a fresh tab is independent (the SSE route calls `listKernels()` on connect), so a deduped-away duplicate is safe. Overridable debounce window: `CELLAR_KERNEL_STATUS_DEBOUNCE_MS` (default 80ms). Startup injections (matplotlib-inline + DataFrame formatter + `%restart_python` magic + project-root `sys.path`) are coalesced into ONE silent `initKernel` exec, not four serial round-trips. Per-card control-in-flight is tracked in `actingPaths`. The notebook list is the `openNotebooks` prop derived from the shell's open notebook tabs in `+page.svelte`, so it updates live as notebooks open/close - not the server `docs` map, which lazily materializes the default notebook on SSR and would show a phantom; its `active` flag reads the same `activeNotebookPath` the Outline/Search sections use, so every sidebar section agrees on which notebook is active. **`activeNotebookPath` is null when no notebook tab is open** - it falls back to the canonical notebook only while that notebook *has* an open tab, because `notebooksCells` is seeded with its SSR cells and an unconditional fallback would fill Outline/Search from a **closed** notebook's stale snapshot (and dot it in the Kernels list). Null means "no active notebook": those sections render empty, which is the honest answer. The status→badge mapping lives in exactly one place, `src/lib/kernelBadge.js` (`kernelBadgeClass`/`kernelStatusLabel`), imported by both `Sidebar.svelte` and the navbar badge in `Navbar.svelte` so the two cannot drift. Only `idle` may render green: `getKernelInfo()` forwards the kernel's status verbatim and the Jupyter status set also includes `restarting`/`autorestarting`/`terminating`/`unknown`, none of which mean "usable". Every card reads its badge from `card.info` through that one mapping, so a live and a not-started card can only differ where meant to. The top-level `/api/kernel` fields (+ the navbar badge / variables / Databricks panel) still track the **active** notebook's kernel via `kernelInfo`; the per-card list is separate. **Kernel-status writes are generation-guarded** (`kernelReqSeq` in `+page.svelte`): `/api/kernel` responses are unordered, so a read issued before the kernel booted can resolve *after* the first run started and clobber `kernelInfo` back to not-started - the last response to *land* would win, not the last one issued. Every write bumps the generation; a response applies only if nothing newer landed while it was in flight. That guard is what lets `onRunEnd` decrement `runsInFlight` **synchronously** rather than awaiting a status round-trip (which would strand the counter if the refresh threw). `onRunStart` asserts the kernel started optimistically via `markKernelStarted()`, which also bumps the generation to supersede in-flight reads - so the card never flashes not-started while a first run boots the kernel), **Databricks** (`databricksSection` -> `src/lib/Databricks.svelte`; mounted lazily on first open, then kept mounted+hidden so collapsing does not discard the connection / cluster list / half-expanded catalog tree - and so a user who never opens it never pays for its python status probe), **Connect an agent** (`agentSection` in `Sidebar.svelte`, reframed around the zero-config flow: leads with an agent-ready banner when a project `.mcp.json` registers the `cellar` stdio server — detected server-side in `+page.server.js` load via `mcpConfigPath`, surfaced as `data.mcp.projectConfigured`, with a `--no-mcp-config` fallback message; shows the one-time `claude mcp add cellar -- cellar mcp` command with a copy button; demotes the live raw `/mcp` Streamable-HTTP URL + config snippet under an "Advanced · raw endpoint" disclosure. The raw URL still comes from `data.mcp.url` reading `CELLAR_MCP_PORT`), **Outline** (one row per heading, nested by level, from the shared `headings.js` `outlineRows(cells, foldedIds, counts)` - so it lists exactly the headings the notebook draws chevrons for). **Every** row has a fold chevron (`data-testid=outline-toggle`), not just rows with nested children: a leaf `## Usage` still owns the code cells under it. Clicking it folds that section in the notebook, and a fold made in the notebook shows here - one state, `foldedIds`, reached through the `onToggleFold` prop (see the Collapsible-headings entry). A folded row is tinted and carries an `…N` hidden-cell marker. The old per-outline `localStorage['cellar-outline-collapsed']` state is gone - it was the second, diverging copy), **Variables**, **Search**. Sections render from a persisted order array (`localStorage['cellar-sidebar-order']`) and are **drag-reorderable** via a per-section grip handle (`data-testid=section-drag-<key>`, native HTML5 DnD, no library); each section's open/closed state persists to `localStorage['cellar-sidebar-open']`. The saved order is untrusted input (hand-edited, or written by an older build), so `src/lib/sidebarSections.js` owns both the canonical `DEFAULT_SECTION_ORDER` and the pure `reconcileSectionOrder()` that merges it with the defaults: unknown, non-string, and **duplicate** ids are dropped (a duplicate would otherwise reach the keyed `{#each}` and throw Svelte's `each_key_duplicate`, discarding the user's whole order), and a section absent from the saved order - i.e. one added since the user last dragged - is restored **beside its default neighbour** (after the nearest preceding default sibling that survived, else before the nearest following one), never appended to the end where a new section would sit below the user's entire layout. - **Shell backend.** `src/lib/server/fstree.js` + `src/routes/api/fs/{tree,file}` — workspace listing and file read/write, all path-guarded to the workspace root. `src/lib/server/inspect.js` + `src/routes/api/kernel/variables` — variable inspector; runs a Python probe through `execute()` (starts from IPython `user_ns` minus `user_ns_hidden`, drops modules/functions/classes + typing/magics scaffolding). `src/routes/api/kernel` + `getKernelInfo()` (added to `kernel.js`) — read-only kernel status that never boots a kernel. Variables refresh after each run and on mount (only if a kernel already exists).
- **Kernel controls (per notebook).** Each kernel card has **Interrupt / Restart / Shut down** buttons (`data-testid=kernel-interrupt`/`kernel-restart`/`kernel-shutdown`), disabled until that notebook has a live kernel. They POST `src/routes/api/kernel/{interrupt,restart,shutdown}` with `{path}` (the notebook, resolved via `resolveNotebookPath`), which call `interruptKernel(nb)`/`restartKernel(nb)`/`shutdownKernel(nb)` in `kernel.ts` — restart is the same restart-over-same-session path the MCP interface uses (namespace clears, process/entry kept, doc survives = "wipe from memory"), while **shutdown** (`teardownKernel` → removes the Map entry) terminates the process and drops the card; the notebook lazily gets a fresh kernel on its next run. Other notebooks are untouched. The page drops stale inspector rows only when the **active** notebook's kernel is restarted/shut down. `listKernels()`/`shutdownKernel()` are covered by `tests/unit/kernel-manager.test.ts`.
- **Kernel lifecycle & memory (Phase 6 - completes the kernel-per-notebook migration).** With N kernels = N Python processes (100s of MB each with pandas/pyspark), three safety nets keep RAM bounded. (1) **Idle culling** leans on the sidecar's own `MappingKernelManager` culler, configured in `bin/cellar.js` `cullingArgs()`: `CELLAR_KERNEL_IDLE_TIMEOUT` (default **7200s/2h**, `0` disables) + `CELLAR_KERNEL_CULL_INTERVAL` (default `min(300, timeout)`). **`cull_connected=True` is load-bearing** - Cellar holds a persistent websocket to every kernel, so with jupyter's default (`false`) an idle kernel is "connected" and never culled. Node-side reconciliation: `kernel.ts` `getManager()` connects the `KernelManager.runningChanged` poll (~10s) to `reconcileCulledKernels()`, which tears down any Map entry whose connection vanished from the server's running list (the manager also disposes that connection itself). A culled kernel therefore drops its sidebar card (`publishKernelStatus`) and bumps its epoch, so its cells read `not_run`/`ran_this_session:false` - the run-status doctrine stays honest. (2) **Soft cap**: `src/lib/kernelCap.ts` (`parseMaxKernels`/`isOverKernelCap`, `DEFAULT_MAX_KERNELS=8`) drives a **warn-only** banner in the Kernels sidebar past `CELLAR_MAX_KERNELS` live kernels (`0` disables) - it **never blocks a run** (the Nth run still lazily starts its kernel); surfaced via SSR `data.maxKernels`. (3) **Shutdown-on-delete**: `shutdownKernelsUnder(path)` (in `kernel.ts`) frees the kernel process of a deleted notebook (or every notebook under a deleted folder), wired into `src/routes/api/fs/op/+server.js`'s `delete` case beside `dropDocs`; idempotent. All three route through **`teardownKernel`**, now idempotent (removes the Map entry synchronously up front) with an `{alreadyGone}` opt-out of the REST shutdown for an already-culled kernel, and it publishes a per-notebook **`kernel:shutdown`** event (`publish`, has `nb`+`seq`) so `LiveNotebook` refetches staleness/run-status (cells read "not run this session"); manual shutdown + `rebindKernel()` fan-out (venv change → tears down EVERY live kernel + clears each queue) emit it too. Tab close does NOT shut a kernel down (keep-alive until cull/manual), and kernels are NOT pre-warmed on tab open (both deliberate - lazy-start's memory win). Covered by `tests/unit/kernel-lifecycle.test.ts` (cull reconciliation, shutdown-on-delete, rebind fan-out) + `tests/unit/kernel-cap.test.ts`.
- **Modal keyboard + shortcut registry.** Jupyter-style command/edit modes. **`src/lib/shortcuts.svelte.js` is the single source of truth**: `DEFAULT_SHORTCUTS` declares every binding as `{id, keys, mode:'command'|'edit'|'global', description, category}`, and the `shortcuts` store merges user rebindings (`localStorage['cellar-shortcuts']`, only overridden ids stored, so a later change to a default reaches every user who never touched it) over them. Bindings are *chord* strings in CodeMirror's syntax (`Mod-Shift-ArrowUp`, `Shift-Enter`, `j`), produced from an event by `chordFromEvent` and rendered for display by `chordTokens`. A binding may also be a **space-separated chord sequence** (`d d` = delete cell): the dispatcher holds the leading chords as a pending prefix (`SEQUENCE_TIMEOUT_MS`, resolved by `resolveChord`) and fires only on the full match, so a lone `d` does nothing; the prefix is discarded the moment the mode changes, which is what keeps it from leaking into an editor. Sequence length is unconstrained, so Jupyter's `i i` / `0 0` need no new machinery. `bindingsCollide` treats a prefix-vs-exact pair (`d` vs `d d`) as a conflict, since the shorter would always win. A chord names the **unshifted** key (`Ctrl-Shift--`, as JupyterLab writes it): `chordFromEvent` reads the key token off `e.code` when Shift is held together with a real modifier (`SHIFTED_BASE_KEY`), because the browser reports `_` for that keystroke. Split-cell is the one platform-conditional entry (`Ctrl` on macOS, `Mod` elsewhere - Jupyter means the literal Ctrl key on both). **Adding a shortcut = a registry entry + an action in `LiveNotebook`'s `actions` map; nothing else may bind a notebook key** (Cell's CodeMirror deliberately carries no run/escape keymap - it was removed so there is exactly one dispatcher). `LiveNotebook.onKeydown` is that dispatcher: one window capture-phase listener, registered only while that notebook is the active tab, so it beats CodeMirror + the browser. It reads the keystroke's mode **off the DOM** (`target.closest('.cm-editor')` -> edit, else command), never off the `keyMode` state - that is what guarantees a command-mode letter (`j`, `a`) can never fire while the user is typing; it also bails on `.modal-open`, on form fields, and on targets outside this notebook's `rootEl`. `keyMode` (visuals only) is driven purely by the editor's own focus/blur, so the indicator cannot drift from the DOM. Each `Cell` hands up an imperative API via `onRegister(id, {focus, focusCell, blur, enterEdit, editorOverlayOpen, run, isMarkdown, currentSource, cursorOffset, replaceSource})` (this replaced the old `focusers` registry); `run` goes through `doRun` so a shortcut always uses the editor's *live* text, not the 500ms-debounced `cell.source`. Cut/copy (`x`/`c`), the heading keys and split-cell compose the last three primitives: `currentSource` is the live editor text, and `replaceSource` rewrites the doc through the remote-apply path (cancels the debounce, suppresses the echo) leaving the **caller** to persist - never call one without the matching `editCell`. The selected cell shows a mode-colored gutter bar + ring + `cmd`/`edit` badge (`info` = command, `success` = edit - semantic tokens, not hardcoded hues, so every theme stays coherent). A newly activated notebook tab focuses its `rootEl` (`tabindex=-1`) so command keys work without first clicking a cell. `keyMode` and the `onRegister`/`onEditorFocus`/`onEditorBlur` props must stay wired all the way down (`LiveNotebook` → `Notebook` → `Cell`): drop any one and the whole modal keyboard goes silently dead, since `cellApis` never fills and the mode never leaves `command`. Fold shortcuts (`h`/Left, `l`/Right) reuse `toggleFold`, and `j`/`k` skip cells a folded heading hides.
  **Focus must follow the selection.** Because the dispatcher reads a keystroke's mode *and* its target off the focused element, a selection the focus doesn't track is a selection the next key doesn't act on (it would reach whatever button or rendered-markdown body the user last clicked). So the cell card itself is focusable (`tabindex=-1`, `outline-none`; its ring already marks selection) and every command-mode selection change goes through `LiveNotebook.selectAndFocus` -> `Cell.focusCell()`: `j`/`k`, `a`/`b`, a command-mode move, Escape out of an editor, `Shift+Enter`'s advance, and a **local** delete (a remote/agent delete only moves the selection - it must not yank the caret out of a cell this user is typing in). `Cell.focus()` means "advance into this cell": its editor, or the cell itself when it is rendered markdown (that editor is `display:none` and cannot take focus). `toggleFold` likewise reselects the header when a fold would hide the selection - command mode may never act on a cell the user cannot see.
  **Cut / copy / paste / undo-delete.** The clipboard (`src/lib/cellClipboard.js`) is a plain module singleton shared by every notebook in the tab (copy here, paste there - Jupyter's behavior), deliberately **not** the system clipboard; an entry is `{cell_type, source, output_scrolled}` and carries **no outputs**, since a pasted output would be a saved result the live kernel never produced. Undo (`z`) is a per-notebook, **local-only** stack: it records what *this* user deleted, so it can never resurrect a cell an agent (or another tab) removed. Paste/undo/split all go through `insertCellAt`, which seeds the new cell server-side via `addCell(afterId, cellType, nb, originId, source)` - one persist, one `cell:added` event carrying the real text - and appends-then-hoists for an insert at index 0 (the add API can only insert *after* an id). `deleteCell` enforces the notebook's **≥1 cell** invariant (the same one the toolbar's delete button enforces by disabling at one cell), so `dd` and `x` are no-ops on a lone cell rather than leaving nothing to type in.
  **Escape is CodeMirror's first.** The dispatcher hands Escape back to CM whenever CM has an overlay of its own open (`Cell.editorOverlayOpen()` = `completionStatus() != null || searchPanelOpen()`), so Escape closes the completion tooltip or the search panel before it ever exits to command mode - Jupyter's behavior. Keystrokes inside a `.cm-panel` (the search bar) are CM's entirely. This is why `@codemirror/autocomplete` and `@codemirror/search` are **direct** dependencies, not just transitive ones under `codemirror`.
  **`Shift+Enter` is one path for both cell types**: `Cell.doRun(advance)` renders a markdown cell / runs a code cell, then `runAndAdvance` selects the next (creating one at the end) and moves focus there - into its editor from edit mode, onto the cell from command mode. A markdown cell never re-enters its own editor on `Shift+Enter`.
  `Settings.svelte` renders the registry grouped by category and rebinds a chord in place, warning on same-mode collisions. A chord that would **type a character** (no modifier but Shift; `typesACharacter` in the registry) bound to an `edit`/`global` shortcut makes that character untypable in every cell: Settings **warns and confirms, but never refuses** (`shortcut-hazard-confirm`, then a standing `shortcut-hazard-warning` on the row). The freedom is deliberate; the surprise is what's refused.
- **Notebook surfaces are three tokens, not per-component classes.** `--cellar-surface-page` / `--cellar-surface-cell` / `--cellar-surface-output` (+ `--cellar-cm-bg` for the editor interior) carry them, applied as `bg-(--cellar-surface-*)` on `Notebook.svelte`'s page wrapper, the `Cell.svelte` card, and its output block. Each token's dark side resolves to the daisyUI surface that was already in use (`base-200` / `base-100`), so dark is untouched by construction. Change a surface here, never with a `bg-base-*` override on a component. **In light themes the page is the greyest plane and every plane above it gets lighter**: grey page → white cell card → faintly-tinted editor → white output. The page grey and the editor grey are the way round they are on purpose; inverting them (the innermost surface darkest, as the first cut of this palette had it) makes a cell read as a hole punched in the page rather than a card resting on it.
- **The cell editor is flush to the card's content edge.** `Cell.svelte`'s `editorEl` wrapper carries no horizontal padding, so `.cm-gutters` starts at the card's inner edge and the input area spans its full width, Jupyter-style. The editor's vertical rhythm is `.cm-content`'s `padding: '10px 0'`; a side inset belongs in neither place.
- **Fluid notebook width.** `Notebook.svelte` content column is `max-w-[clamp(48rem,92%,88rem)]` — grows with the viewport up to a readable 88rem cap (not full-bleed on ultrawide).
- **File tree UX (VS Code-style).** Folders start **collapsed** (`FileTreeNode` `open=false`), only root visible. Entries carry self-contained inline-SVG **file-type icons** (`src/lib/fileIcons.js`, keyed by extension, no CDN/network — offline-safe). **Git decorations**: `src/lib/server/git.js` + `src/routes/api/fs/git` return `git status --porcelain -uall` as `{isRepo, files:{relPath→letter}}`; the tree colors names + shows a status letter (M gold, A/U green, D/C red) and rolls child status up to collapsed folders (a dot). Refreshed on mount, manual refresh, window focus, and after file saves (`fsRefreshSignal` bump); a non-git workspace returns `isRepo:false` → clean tree, no errors.
- **Git change decorations below the file level (line + cell).** The tree says *which files* changed; these say *which lines* (editor gutter bars) and *which cells* (notebook accent bars). **Baseline, not diff:** `gitHeadFile()` (`git.js`) + `GET /api/fs/git/head?path=[&kind=notebook]` hand the browser the file's git-HEAD text **once** (the `kind=notebook` form parses it through `ipynb.js`'s `deserialize`, so both sides of a cell diff are normalized identically and nbformat's multiline `source` arrays never read as a change); the client re-diffs the live buffer against it. That is what keeps markers live per keystroke without a `git` process per keystroke — do NOT "optimize" this into a server-side `git diff` route. `tracked:false` (not a repo / no commits / untracked / binary / unparseable) ⇒ **no decorations at all**, matching VS Code: the tree's `U`/`A` letter is already the whole story. Baselines refetch on mount, on `fsRefreshSignal`, and on window `focus` (a commit made in a terminal).
- **`src/lib/gitdiff.js`** is the shared, pure, browser-safe engine. `lineChanges()` = Myers' greedy line diff (common prefix/suffix trimmed first; bails to one whole-region hunk past `MAX_EDIT_DISTANCE`), returning per-line `add`/`mod` plus deletion seams. `notebookCellChanges()` matches cells **by id, never by index** — Cellar owns nbformat 4.5 ids, so an id survives inserts and reorders, whereas index matching would paint every cell below an insert as modified (a HEAD notebook with missing/duplicate ids falls back to index matching). It compares `source` + `cell_type` only: **outputs are deliberately excluded**, so re-running a cell to the same result never lights up a bar. A pure reorder is flagged `moved` on the *minimal* set of cells (the complement of the LIS of HEAD positions), so dragging one cell up marks one cell, not every cell it jumped. A removed cell has no cell to decorate, so its marker anchors to the next surviving HEAD cell (`removedBefore`) or falls off the end (`removedAtEnd`), rendering as a dashed seam in `Notebook.svelte`.
- **`src/lib/gitGutter.js`** is the CodeMirror half (`FileTab.svelte` only; notebook cells use the `Notebook.svelte` bars instead). Its `gutter()` is listed **after `basicSetup`** so the bar is the rightmost gutter, hard against the code. Two sharp edges: `highlightActiveLineGutter` classes the caret's line in *every* gutter, so `.cm-gitGutter .cm-gutterElement` must force `background: transparent` and the change bars must be `::before` overlays — otherwise the active line shows a phantom bar; and the deletion triangle uses `::after`, so one line can be both changed and a deletion seam.
- **Git decoration colors are NOT daisyUI semantic tokens.** Those are accent colors and their hues are not stable across themes (`dim` makes `secondary` and `error` both salmon; `nord` makes `info` pink), so added/modified/removed would stop reading as added/modified/removed. They are fixed palettes in `app.css`, resolved per scheme by CSS `light-dark()` against the `color-scheme` daisyUI sets on `<html data-theme>` — the same source of truth `editorTheme.js` uses, so every future theme is handled with no per-theme work. There are **two** sets, because the two surfaces follow different conventions: `--cellar-git-*` for the diff bars (`gitGutter.js`, `Notebook.svelte`: added green / modified **blue** / moved purple / removed red), and `--cellar-git-tree-*` for the file explorer's names and status letters (`FileTreeNode.svelte`: untracked+added green / modified **gold** / deleted red, VS Code's explorer convention). Both sides of each `light-dark()` are that scheme's own value — VS Code's *dark* decoration hues sit near 1.5:1 on a light background, which is what made the tree's green unreadable.
- **Git blame in the bottom status bar (GitLens-style).** For the active **file tab's** cursor line, the shell footer (`+page.svelte`) shows "author, relative date" with the commit short-sha + summary on hover; uncommitted lines read "You, uncommitted" (hover "Not committed yet"). `gitBlameFile()` (`git.js`) runs `git blame --line-porcelain -w` and parses one record per line (`{commit, shortSha, author, authorTime, summary, notCommitted}`; the all-zero SHA means `notCommitted`); route `GET /api/fs/git/blame?path=` returns `{isRepo, tracked, lines}`, `tracked:false` (untracked/binary/non-repo) hides the bar. **Whole-file cache, not per-keystroke git:** `FileTab.svelte` fetches the blame array once (on mount, `gitRefresh` bump, window focus, and after its own save) and maps the cursor line to a record on a 90ms-debounced `selectionSet`/`docChanged`, reporting up via `onBlame(path, record|null)` into `+page.svelte`'s `blameByPath` map; the footer reads `activeFilePath`'s entry. **Notebooks are deliberately out of scope** - blaming `.ipynb`/`.py` JSON lines doesn't map to cell source lines, so `LiveNotebook`/`Cell` never report and the bar is empty for a notebook tab. Relative dates use `relativeTimeLong()` in `relativeTime.js` (weeks/months/years horizon, separate from the run-badge's day-capped `relativeTime`). Footer text is a daisyUI semantic token (`text-base-content/55`), theme-agnostic by construction - unlike the diff-bar colors, blame is neutral chrome, not an add/mod/del signal.
- **File management (VS Code-style explorer).** Create/rename/delete/cut-copy-paste (move/copy) for files + folders. **Backend**: `fstree.js` exports `createEntry`/`renameEntry`/`deleteEntry`/`moveEntry`/`copyEntry` (folders recursive via `cpSync`/`rmSync`), all path-guarded through `resolveInWorkspace` + a `resolveInside` helper that also refuses the workspace root; single-segment name validation (also rejects `.`-prefixed hidden names, which the explorer never lists); moving/copying a folder into itself/a descendant is refused and a move within the same parent is a no-op; paste auto-suffixes on collision (` copy`, ` copy 2`, …), create/rename reject on collision. One dispatch route `POST /api/fs/op` (`{op, …}`) returns the affected workspace-relative path(s). Delete/rename/move also sync the live notebook doc map server-side (`+server.js` calls `notebook.js`'s `dropDocs`/`rekeyDocs`): a deleted path (or any doc nested under a deleted folder) is dropped from the `docs` Map so a later UI/MCP persist can't `writeFileSync`-resurrect the removed file, and a renamed/moved path is rekeyed to its new absolute id (folder ops rekey nested notebook docs too) so edits keep landing in the live doc; the `activePath` pointer follows (reset to null → default notebook when a dropped doc was active). **Frontend**: `Sidebar.svelte` owns the file-ops state (clipboard, inline rename/new, context menu, delete-confirm modal) and exposes it to the recursive `FileTreeNode.svelte` via the `cellarFileOps` **context** (getters forward `$state` so nodes stay reactive — avoids prop-drilling). Inline rename + new-entry use `TreeEntryInput.svelte`. Right-click any node (or empty files-body / root) → context menu; F2/Delete/⌘·Ctrl+C/X/V shortcuts; a cut entry dims until pasted; Paste only on folders/root. After every op the tree + git decorations refresh, and `onFsChange({type,from,path})` bubbles to `+page.svelte` so open tabs are closed (delete) or remapped (rename/move) — no tab left dangling at a gone path.
- **Tabs — session memory + preview.** Open-tab set is persisted per workspace to `localStorage['cellar-tabs:'+workspace]` (paths + preview flag + active id). **First-ever open** (no key) → empty state (`data-testid=empty-state`, no tabs); returning → tabs restored. Single-click a tree file → transient **preview** tab (one at a time; next single-click reuses its slot); **double-click** (tree or tab) or **editing** promotes it to permanent (VS Code semantics). All tab logic lives in `+page.svelte`.
- **Opening a workspace `.ipynb` (live).** Any `.ipynb` opened from the tree loads as a **live, kernel-attached** notebook via `LiveNotebook` (view/edit/run, edits saved back to that file through clean-on-save) — `ipynb`-kind tabs render `LiveNotebook path={tab.path}`, the canonical one uses tab id `notebook`. Preview/permanent tab semantics are unchanged (single-click preview slot, double-click/edit promotes). The old read-only `NotebookFileView` + `/api/fs/notebook` + `readWorkspaceNotebook` were removed.
