## Current state: MVP Step 1 — notebook document + save pipeline

Built on the Phase 0 bridge spike. Full product/tech spec lives outside the repo at `firstmate/data/cellar-spec.md` (read §3 for the versioning model); nbdev field-policy detail at `firstmate/data/nbdev-study-n8/report.md`. See `README.md` for run/verify details.

- **Two runtimes.** Node (SvelteKit) + a Python sidecar (Jupyter kernel service). Both are booted by the CLI launcher `bin/cellar.js` (one command → both servers up → browser opens, workspace = cwd). Distributed as an **npm package** (`package.json` `files`=`build/`+`bin/`+`src/lib/server/venv.js`; `prepack`/`prepublishOnly` run `vite build`); launcher defaults to the prod build (`node build/index.js`), `--dev` uses Vite. `build/`/`.venv/` gitignored. A root **`Makefile`** wraps these mechanics for convenience (`make setup` = install+build+`chmod +x bin/cellar.js`+`npm link`; `make update` = `git pull`+install+build; plus `make build`/`run`/`dev`; bare `make` prints help) - it invents no new scripts, just the real npm/launcher commands. `bin/cellar.js` is committed **executable** (git mode 100755 - the shebang'd, sole `package.json` `bin` entry); preserve that bit if you ever regenerate it, or `npm link` yields a non-runnable `cellar` symlink (the `make setup` `chmod` is a belt-and-suspenders re-set).
- **Per-repo launcher + venv binding (`bin/cellar.js` + `src/lib/server/venv.js`).** `cellar` in any repo binds the kernel to the **project's** venv. Resolution (first match): `--venv`/`CELLAR_VENV` → `$VIRTUAL_ENV` → `<ws>/.venv` → create. All venv create/installs go through **`uv`** (`uv venv`, `uv pip install`) — hard requirement, fails fast if absent (no `python -m venv` fallback). Ensures `ipykernel` in the project venv; **confirms before** create/install on a TTY (`--yes`/`-y`/`$CI`/non-TTY auto-approve; `--python` = bind arbitrary interpreter, no create/install). `venv.js` uses node builtins only so it's importable by both the launcher (`../src/lib/server/venv.js`) and the SvelteKit server (`$lib`). Cellar's heavy Jupyter host env (`jupyter-server`) lives in a private `~/.cellar/host-venv`, created/cached first run, kept separate from project venvs.
- **Kernel bridge = the de-risked core.** `src/lib/server/kernel.js` connects to Jupyter over REST+WebSocket using `@jupyterlab/services` (spec §2). Node 18+ global `fetch`/`WebSocket` are passed into `ServerConnection.makeSettings` — no polyfill. `execute()` emits real nbformat output objects so the same shape streams live to the browser and is persisted. `rebindKernel()` (added) tears down + restarts the kernel so it re-reads a rewritten kernelspec onto a new interpreter (a plain `restart()` reuses the original argv and would NOT switch venvs). **Startup injection (`initKernel`):** every kernel silently runs `%matplotlib inline` on fresh start **and** after `restartKernel()` (restart clears the namespace + backend), so matplotlib figures emit `image/png` even when the project's default backend isn't inline (e.g. `MPLBACKEND=agg`/macosx); guarded by try/except so it's a harmless no-op when matplotlib is absent.
- **Kernelspec = the venv lever.** The launcher writes a per-run `python3` kernelspec (`writeKernelspec`) whose `argv[0]` is the project python, into a temp `JUPYTER_PATH` dir handed to the sidecar; `startNew({name:'python3'})` then launches the project interpreter — `kernel.js` unchanged. The dir is passed to the app as `CELLAR_KERNELSPEC_DIR` so the runtime venv control can rewrite it.
- **Sidecar cwd must equal `root_dir` (= WORKSPACE).** `bin/cellar.js` spawns the Jupyter sidecar with `cwd: WORKSPACE`, not Cellar's `REPO`. Because `startNew({name:'python3'})` passes no notebook path, `jupyter_server` doesn't derive a kernel cwd from `root_dir` and the kernel inherits the sidecar's process cwd — so anchoring it at WORKSPACE is what makes `os.getcwd()`, relative reads/writes, and repo-root walks resolve in the user's project (else they resolve in / write into Cellar's install dir). Everything else in that spawn (host python, `JUPYTER_PATH`) is already an absolute path, so it stays REPO-independent. **Follow-up (not done):** for multi-directory workspaces, pass each notebook's workspace-relative path to `startNew(...)` so a kernel roots at its own notebook's folder — deferred because a single kernel is shared across all open notebooks, making a per-notebook root ambiguous.
- **Runtime venv control.** Settings → "Python environment" (`src/lib/Settings.svelte`) → `src/routes/api/venv` (GET info / POST `{path,create}`) → `src/lib/server/venv-bind.js`: resolves/creates the venv via uv, ensures ipykernel, rewrites the kernelspec, updates in-process `CELLAR_PROJECT_VENV`, then `rebindKernel()`. Confirm-before-create is enforced in the UI.
- **Notebook documents (server-owned, multi-doc).** `src/lib/server/notebook.js` holds in-memory notebooks keyed by absolute path in a `docs` Map — the default `notebook.ipynb` **and** any other workspace `.ipynb` opened live. Every exported op takes an optional trailing `nb` (workspace-relative or absolute path); omitting it targets the **active** notebook (`activePath`). `setActiveNotebook(nb)`/`getActiveNotebookPath()` track which notebook the agent-facing (MCP) tools default to; the UI POSTs `/api/notebooks` to follow tab focus. **Loading never writes to disk** (`loadDoc`): the default notebook is materialized *in memory* if missing so callers get a valid shape (SSR seeds the shell from `getDefaultNotebook()`), but no `.ipynb` is written on a bare startup/open — so opening Cellar in a folder drops **no** uninvited file. Every notebook (default included) persists only on a genuine mutation (add/edit/run/…) or an explicit `createNotebook`, which materializes the file (and the UI's empty-state "New notebook" button → `POST /api/notebooks {create:true}` → `createNotebook`); an arbitrary opened `.ipynb` must exist and is likewise NOT rewritten until mutated (no surprise git diff on open). Startup with no `notebook.ipynb` shows the shell's empty state (`+page.svelte`, `data-testid=empty-state`). Cellar owns cell IDs (uniqueness enforced **per document** on load via `enforceUniqueIds`; ids may legitimately repeat across notebooks — do NOT assume global uniqueness).
- **Persistence + clean-on-save.** `src/lib/server/ipynb.js` (de)serializes canonical doc ↔ real nbformat 4.5 with **deterministic** output (fixed key order, 1-space indent) so identical re-runs produce zero git diff. `src/lib/server/clean.js` is the nbdev field-policy port: nulls all `execution_count`, deny-by-default metadata allowlist (cell keeps only the `cellar` namespace; notebook keeps only `kernelspec` — drops `language_info`/`widgets`), normalizes `kernelspec.display_name`→`name`, scrubs `<… at 0x…>` memory addresses. Must stay **idempotent**.
- **API.** `+page.server.js` load returns the default notebook + seeds the sidebar. `src/routes/api/notebooks` (`GET ?path=` loads a live doc's cells; `POST {path}` sets the active notebook, or `POST {path,create:true}` materializes it on disk via `createNotebook`). Cell ops under `src/routes/api/cells/…` (`POST` add, `PATCH` source/type, `DELETE`, `POST …/run` streams NDJSON + persists, `…/move`, `…/clear`) all carry an `nb` path (body field, or `?nb=` query for clear/delete) so they mutate the addressed notebook, not just the active one.
- **Cell types.** Code and **Markdown** (nbformat `cell_type`). Markdown cells carry source + no outputs/execution_count; clean-on-save + UUID ids apply the same. `addCell(afterId, cellType)` and `setCellType(id, type)` (PATCH `cell_type`); converting to markdown clears outputs.
- **UI.** `src/lib/Cell.svelte` = one cell — CodeMirror editor (language via a Compartment: python / markdown; editor theme via a second Compartment that follows the app theme; `theme` prop drilled shell→`LiveNotebook`→`Notebook`→`Cell` (and shell→`FileTab`)). **Editor theming is centralized in `src/lib/editorTheme.js`** (`editorThemeExtensions(theme)` + `isLightTheme(theme)`), shared by `Cell.svelte` and `FileTab.svelte`: light vs dark is derived from the **resolved `color-scheme`** daisyUI sets on `<html data-theme>` (light themes get the pygments-"default" Jupyter light scheme on a soft-grey editor surface `LIGHT_EDITOR_BG` `#f6f6f6` - applied to the surface, `.cm-content`, and a borderless `.cm-gutters` - so the code area stays distinct instead of stark white; dark get `oneDark`, deliberately untouched), not a hardcoded theme-name allowlist, so every light theme is handled automatically (the `LIGHT_THEMES` name set is only an SSR/pre-DOM fallback). The reconfigure `$effect` must read `theme` **outside** the `if (view)` guard - `view` is assigned in `onMount`, which can run after the effect's first pass, so a guarded read never registers `theme` as a dependency and the editor stays stuck at its mount-time theme on toggle. `Cell.svelte` also carries the per-cell toolbar and renders nbformat outputs. **Image outputs** (`image/png`/`jpeg`/`gif` base64, `image/svg+xml` URI-encoded) render as an `<img data-testid=output-image>` via `imageDataUrl()`; the image mime is **preferred over `text/plain`** so a matplotlib figure shows the plot, not its `<Figure … with N Axes>` repr (the kernel's inline-backend injection is what makes `image/png` present - see the Kernel bridge entry). Markdown cells render to sanitized HTML (markdown-it `html:false` + DOMPurify, client-only via `browser` guard); double-click / pencil button → raw edit; Run/⌘Enter renders in place, Shift+Enter renders+advances. Type toggle is the far-right `python3`/`markdown` label. Per-notebook cell ops (add code/markdown, delete/reorder, single-kernel one-run-at-a-time; markdown "run" persists source only, no kernel) live in `src/lib/LiveNotebook.svelte`, which owns one notebook's cells + renders via `Notebook.svelte` (pure renderer). `Cell` seeds its editor doc once on mount and does NOT sync later `cell.source` changes into the editor — so `LiveNotebook` fetches authoritative cells on mount (never pre-seeds a stale snapshot). Editor source autosaves via debounced `PATCH`. Dev exposes `window.cellarViews[id]`. **Active cell** = `LiveNotebook` `activeId` (set on a Cell's `onfocusin`/`onpointerdown`, and on keyboard move); the active Cell gets a `border-primary` ring + a left accent bar (`data-active`), works in both themes. **Drag-to-reorder cells**: a per-cell grip handle (`data-testid=drag-handle`, `draggable`) + HTML5 DnD drop zones in `Notebook.svelte` → `onMoveToIndex` → the move route's `{toIndex}` → server `moveCellTo` (persists + git-clean like any move; keyboard ⌘/Ctrl+Shift+Arrow still works). **Scrollable outputs** (Jupyter "Enable Scrolling for Outputs"): a per-cell toggle contracts a tall output to a fixed-height scroll box; auto-applies above `SCROLL_THRESHOLD` px unless overridden. The choice persists in `metadata.cellar.output_scrolled` (undefined=auto / true / false) — round-trips through clean-on-save; note `cellView` in `notebook.js` now carries `metadata` so the browser sees it, set via PATCH `{scrolled}`. **Collapsible code editor** (sibling of scrollable-outputs, code cells only): the editor grows with content up to `EDITOR_MAX_PX` (576px = `max-h-[36rem]`), then a per-cell toggle (`data-testid=editor-collapse-toggle`, same style/placement as the outputs-scroll toggle) contracts it to that fixed scrollable box; auto-caps tall editors unless overridden (tri-state undefined=auto / true / false; a `ResizeObserver` on `editorEl` measures unclamped `scrollHeight`, covering content growth + the hidden→visible tab transition). Unlike `output_scrolled`, this choice is **runtime-only** like fold state — `LiveNotebook` `editorCollapsed` map persisted to `localStorage['cellar-editor-collapsed:'+absPath]`, **never** written to the `.ipynb` (zero git-diff). Markdown edit views keep the original `max-h-96` cap. **Markdown tables in text/stream output** are parsed (header + `|---|` separator) and rendered as sanitized daisyUI `table table-zebra table-xs` (same markdown-it + DOMPurify path as markdown cells); non-table text is unchanged. **Collapsible headings** (Jupyter "Collapsible Headings"): a markdown header cell (`#`…`######`) gets a chevron (`data-testid=fold-toggle`) that folds its section - every cell until the next header of the **same or higher** level (nested folds compose: an H1 folds its H2/H3 sections too). Section boundaries + header level come from the shared `src/lib/headings.js` (`headerLevel` + `computeFolding`) - the same header logic the sidebar Outline and MCP `get_notebook_map` use, so reuse it rather than re-parsing headers. Folded cells stay in the doc (run/persist normally); `Notebook.svelte` just `class:hidden`s them and `Cell.svelte` shows an "N cells hidden" hint. Fold state is **runtime-only** (`LiveNotebook` `foldedIds`, persisted to `localStorage['cellar-folds:'+absPath]`, restored on load) so it never touches the `.ipynb` (zero git-diff) - the deliberate contrast with `output_scrolled`, which does round-trip through `metadata.cellar`.
- **MCP agent interface.** `src/lib/server/mcp/` — the agent surface (spec §4). `service.js` = transport-independent tool logic (section tree from markdown headers, tiered/summarized outputs, visibility filtering, UUID-addressed read/write/execute); `server.js` = `McpServer` + Streamable HTTP transport on its own port (`CELLAR_MCP_PORT`, default 39587 but the launcher now allocates a **free** port per instance so concurrent `cellar` runs don't collide, path `/mcp`), started once from `src/hooks.server.js`. Runs **in-process** in the SvelteKit backend so it shares the live `notebook.js` doc + `kernel.js` — restarting the kernel can't drop the MCP session/document (the #1 requirement). Per-cell hide/show via `metadata.cellar.hidden_from_agent` (allowlisted, visible by default) is honored in every map/read/search/section/result. Kernel service methods `restartKernel/interruptKernel/kernelStatus` live in `kernel.js`; richer read/write (`listCells/getCell/setVisibility/moveCellTo`) in `notebook.js`. **Coherence doctrine:** `server.js` passes an `INSTRUCTIONS` house-style string as the `McpServer` `instructions` (advisory, delivered once at connect: imports-at-top convention, check `kernel_state` first, continue the story via `get_notebook_map`, structure with markdown) + a matching `cellar_notebook_style` prompt. The read-group **`kernel_state`** tool returns the live namespace bucketed into imports/functions/classes/variables (types, shapes, dataframe columns); it reuses `inspect.js`'s shared probe (`kernelState()`), short-circuits to `{started:false}` when `kernelStatus()` is `not_started` (no force-boot) and to `{started:true,busy:true}` when the kernel is busy (never queues a probe behind a running cell), and dedupes imports by statement. `inspectVariables()` (sidebar) shares that probe but projects to the original `{name,type,shape,preview}` shape.
- **Zero-config agent connection (`cellar mcp`).** Because the MCP HTTP port is per-run dynamic, agents are pointed at the stdio command `cellar mcp`, never a URL. On launch `bin/cellar.js` writes `<ws>/.cellar/runtime.json` (`{mcpPort,appPort,jupyterPort,pid}`, ephemeral - gitignored, cleared in `cleanup()` on shutdown) and writes/merges `<ws>/.mcp.json` with a `cellar` stdio entry (`--no-mcp-config` opts out). Both helpers live in `src/lib/server/runtime.js` (node-builtins + `fetch` only, importable by launcher and `$lib`); `.mcp.json` merge is idempotent and preserves other servers. The `mcp` subcommand (handled at the top of `bin/cellar.js`, before normal arg parsing) calls `runMcpBridge` in `src/lib/server/mcp-bridge.js`: reads runtime.json, verifies alive (pid AND a GET to `/mcp`), then transparently proxies JSON-RPC both ways between a `StdioServerTransport` and a `StreamableHTTPClientTransport` to the live server (schema-agnostic, so it never drifts). Missing/dead instance → clear stderr error + exit 1 (never auto-launches). `runMcpBridge` returns a promise that stays pending until shutdown - the launcher must `await` it (an early `process.exit` kills the bridge right after connect). Both new modules are added to `package.json` `files`. The sidebar "Connect an agent" panel is reframed around this flow (see Sidebar entry below).
- **Live server→client run feedback (SSE, P0+P1).** An open browser reflects *agent* (or other-tab) runs with no reload: running indicator + streaming outputs. `src/lib/server/events.js` = in-process `EventEmitter` pub/sub (`publish`/`subscribe`); it stamps each event with a per-notebook monotonic `seq`. `src/routes/api/events/+server.js` = one SSE stream per browser tab (`text/event-stream` `ReadableStream`, `hello` + 15s heartbeat + abort/cleanup) carrying events for **all** notebooks. `src/lib/events-client.js` = a **single shared `EventSource` per tab** (ref-counted; stays under the HTTP/1.1 ~6-conn cap) plus a per-tab `originId`; every `LiveNotebook` subscribes to it. Both run entry points publish `run:start`/`run:output` (per chunk)/`run:end`: the UI `/api/cells/[id]/run` route (`actor:'user'`, carries the caller's `originId`) and MCP `service.js` `runCell` (`actor:'agent'`; `runCells`/`runAll`/`runRange` funnel through it). **Canonical notebook id (the P0 gotcha):** events are tagged with the notebook's **absolute** path; `LiveNotebook` learns that id from its `GET /api/notebooks` load response (`body.notebook.path`) and filters SSE on it — the workspace-relative `path` prop is only for the REST API (server resolves it via `resolveNotebookPath`/`resolveAbs`). **Echo:** the initiating tab drops its own events by `originId` match (it renders that run from the `/run` NDJSON response), so a user run never double-renders; agent runs carry no `originId` so every tab renders them. **Backstop:** on `EventSource` (re)connect or a detected `seq` gap, the notebook refetches. Deferred to later phases (NOT built here): unified server-side run function, the busy/queue hang fix (run metadata landed later in P3, see below).
- **Live structural sync (SSE, P2).** Agent-driven (and cross-tab) *structural* changes also reflect live with no reload. **Single chokepoint:** `notebook.js`'s mutation ops (`addCell`/`setSource`/`setCellType`/`deleteCell`/`clearOutputs`/`moveCell`/`moveCellTo`) `emit()` `cell:added`/`cell:edited`/`cell:type`/`cell:deleted`/`cell:cleared`/`cell:moved` (schema per scout report §3.3), tagged with the doc's absolute path + `seq` + an optional `originId` threaded through from the UI REST routes (MCP calls pass none → every tab renders). `LiveNotebook` `applyStructuralEvent` patches its `cells` in place (insert/remove/reorder/retype), gated by the same `originId` echo-suppression + `seq`-gap-refetch backstop as runs. **Editor-safety rule (report §3.4):** a `cell:edited` hands the new source to `Cell.svelte` via `cell.remoteEdit`; the Cell applies it to CodeMirror only when the cell is **not** focused / has no pending local edit — otherwise it shows a "Changed on server" affordance (`data-testid=remote-changed`) with a **Load** button, so remote edits never clobber active typing. Applying a remote edit sets `applyingRemote` so the editor's update listener doesn't echo it back as a local save. **New/active notebooks surface live:** MCP `create_notebook(name?)` really creates (or opens) a workspace `.ipynb`, makes it active, and publishes `notebook:opened {nb,relPath,name}`; the shell (`+page.svelte`) subscribes and opens/focuses a tab via `openFilePermanent` (own-action echo dropped by `originId`). Because the agent's write ops default to `activePath` (the notebook the UI last focused / just created), agent edits land in the exact in-memory doc the open tab shows.
- **Single-instance-per-folder (`bin/cellar.js` step 0 + `runtime.js`).** A relaunch in a folder that already has a live cellar does NOT start a rival - two servers each persist `notebook.ipynb` from independent in-memory docs, so last-writer-wins silently clobbers edits. **`.cellar/instance.lock`** is an `O_EXCL` lockfile (`acquireInstanceLock`) claimed at the very top of `main()` **before** the uv/venv work: it atomically gates folder ownership, which is what closes the boot-window race (runtime.json is written mid-boot, so a pid-alive/port-down probe alone would wrongly read "dead" and clobber - the lock is decided synchronously, before that window). The launcher that loses the lock **attaches**: `waitForInstanceUrl` polls the owner's runtime.json + app port until reachable (covers the owner still booting), opens the browser to it, and `process.exit(0)` without touching anything. A **dead** owner's lock is stale and silently reclaimed inside `acquireInstanceLock`'s retry loop; if the owner vanishes mid-boot, the attach path retries the lock to take over. `--new`/`--force` skips the lock to run a deliberate second instance (does not take/release the lock; overwrites runtime.json last-writer-wins, the accepted power-user tradeoff). `cleanup()` releases the lock (pid-guarded `releaseInstanceLock`) only if we took it. Attaching needs no venv/toolchain, so step 0 runs before the uv check.
- **Editor edit-flush (no-run persistence).** `Cell.svelte` autosaves source on a 500ms debounce; it also `flushEdit()`s immediately on editor **blur** and window **`pagehide`**, and teardown flushes rather than cancelling - so a cell edit survives closing Cellar even inside the sub-debounce window, without a run. `LiveNotebook.editCell(id, source, {keepalive})` sets `keepalive:true` **only** from the unload flush (Cell passes `{keepalive:true}` from `pagehide`); normal page-alive autosaves stay plain `fetch` because the browser caps a keepalive body at ~64KB - routing every autosave through keepalive silently drops edits to cells >64KB (the regression to avoid). PATCH is `.catch`'d so a rejected unload flush never surfaces as an unhandled rejection.
- **Not yet built (deferred by design):** MCP `get_changes_since`/delta reads + subscribable-resource map, extract-to-`.py`, `.py` view, Databricks, git merge driver, per-output size threshold, LaTeX/math + code-block highlighting in rendered markdown, polished editing UX, live-sync later phases (unify UI+MCP run path / run metadata / server-side run serialization + queued-state push).

## Application shell (navbar + sidebar)

Wraps the notebook in a JupyterLab-style shell. `src/routes/+page.svelte` is the shell: it owns tabs/sidebar/kernel-controls/theme and hosts one `LiveNotebook` per open notebook tab (default + each opened `.ipynb`), all kept mounted+hidden so editor/run state survives tab switches. It no longer owns `cells`: each `LiveNotebook` owns its own cells and reports the live array up via `onCellsChange(path, cells)` into a path-keyed `notebooksCells` map; the sidebar (outline/search/variables) reads `activeCells` for the focused notebook (falls back to the default when a plain file tab is active). A single shared kernel → the shell serializes runs across notebooks via a global `runBusy` flag (`onRunStart`/`onRunEnd`).

- **Navbar** (`src/lib/Navbar.svelte`): app-menu dropdown → `src/lib/Settings.svelte` modal (theme toggle `dim`↔`nord`, persisted to `localStorage['cellar-theme']`, applied via `<html data-theme>`); editor-style **wrapping** tab bar (tabs flow onto more rows via `flex-wrap`, never clip). Every tab (including the notebook) is closable and part of session memory; tabs carry file-type icons + a dirty dot; `preview` tabs render italic.
- **File tabs** (`src/lib/FileTab.svelte`): CodeMirror editor over a workspace file, ⌘S save; syntax-highlighted by file extension (`langFor`: python / markdown / json+ipynb / yaml, else plain text), sharing `editorTheme.js` light/dark theming with cells; dirty state bubbles up to the tab bar. Notebook + all file tabs stay mounted (hidden via `.hidden`) so editor/run state survives tab switches.
- **Sidebar** (`src/lib/Sidebar.svelte`, foldable sections; whole sidebar collapsible + **drag-resizable** via the `sidebar-resizer` handle in `+page.svelte`, width persisted to `localStorage['cellar-sidebar-width']`, bounds 180–560px): **Files** (`src/lib/FileTreeNode.svelte`, recursive), **Kernels** (`kernelsSection` in `Sidebar.svelte`: reflects **real runtime state** for the one-shared-kernel model - before the first run it shows a "No kernel running / not started" card, not a phantom `python3`; once booted it shows the single kernel + its live status with **every open notebook** listed under it; a row *focuses* its tab via `selectTab` (never `openFilePermanent`, so clicking a preview notebook does not promote it out of the single-click preview slot) and the active one is dotted. The notebook list is the `openNotebooks` prop derived from the shell's open notebook tabs in `+page.svelte`, so it updates live as notebooks open/close - not the server `docs` map, which lazily materializes the default notebook on SSR and would show a phantom; its `active` flag reads the same `activeNotebookPath` the Outline/Search/Variables sections use, so every sidebar section agrees on which notebook is active. The status→badge mapping lives in exactly one place, `src/lib/kernelBadge.js` (`kernelBadgeClass`/`kernelStatusLabel`), imported by both `Sidebar.svelte` and the navbar badge in `Navbar.svelte` so the two cannot drift. Only `idle` may render green: `getKernelInfo()` forwards the kernel's status verbatim and the Jupyter status set also includes `restarting`/`autorestarting`/`terminating`/`unknown`, none of which mean "usable"), **Connect an agent** (`agentSection` in `Sidebar.svelte`, reframed around the zero-config flow: leads with an agent-ready banner when a project `.mcp.json` registers the `cellar` stdio server — detected server-side in `+page.server.js` load via `mcpConfigPath`, surfaced as `data.mcp.projectConfigured`, with a `--no-mcp-config` fallback message; shows the one-time `claude mcp add cellar -- cellar mcp` command with a copy button; demotes the live raw `/mcp` Streamable-HTTP URL + config snippet under an "Advanced · raw endpoint" disclosure. The raw URL still comes from `data.mcp.url` reading `CELLAR_MCP_PORT`), **Outline** (nested markdown-header tree; a node with children collapses via a per-node chevron, state keyed by heading path in `localStorage['cellar-outline-collapsed']`), **Variables**, **Search**. Sections render from a persisted order array (`localStorage['cellar-sidebar-order']`) and are **drag-reorderable** via a per-section grip handle (`data-testid=section-drag-<key>`, native HTML5 DnD, no library); each section's open/closed state persists to `localStorage['cellar-sidebar-open']`. Section bodies are Svelte `{#snippet}`s dispatched by key so order is data-driven.
- **Shell backend.** `src/lib/server/fstree.js` + `src/routes/api/fs/{tree,file}` — workspace listing and file read/write, all path-guarded to the workspace root. `src/lib/server/inspect.js` + `src/routes/api/kernel/variables` — variable inspector; runs a Python probe through `execute()` (starts from IPython `user_ns` minus `user_ns_hidden`, drops modules/functions/classes + typing/magics scaffolding). `src/routes/api/kernel` + `getKernelInfo()` (added to `kernel.js`) — read-only kernel status that never boots a kernel. Variables refresh after each run and on mount (only if a kernel already exists).
- **Kernel controls.** Sidebar Kernels section has **Interrupt**/**Restart** buttons (`data-testid=kernel-interrupt`/`kernel-restart`), disabled until the kernel is started. They POST `src/routes/api/kernel/{interrupt,restart}`, which call `interruptKernel()`/`restartKernel()` in `kernel.js` — the same restart-over-same-session path the MCP interface uses, so the doc survives while the namespace clears (the page drops stale inspector rows on restart).
- **Move-cell shortcuts.** `LiveNotebook.svelte` `onKeydown` (window capture-phase, registered only while that notebook is the active tab, so it beats CodeMirror + the browser): ⌘/Ctrl+Shift+Arrow moves the focused cell (`closest('[data-cell-id]')`) via `moveCell`, then restores focus through the `focusers` registry so moves chain; no-ops at the edges.
- **Fluid notebook width.** `Notebook.svelte` content column is `max-w-[clamp(48rem,92%,88rem)]` — grows with the viewport up to a readable 88rem cap (not full-bleed on ultrawide).
- **File tree UX (VS Code-style).** Folders start **collapsed** (`FileTreeNode` `open=false`), only root visible. Entries carry self-contained inline-SVG **file-type icons** (`src/lib/fileIcons.js`, keyed by extension, no CDN/network — offline-safe). **Git decorations**: `src/lib/server/git.js` + `src/routes/api/fs/git` return `git status --porcelain -uall` as `{isRepo, files:{relPath→letter}}`; the tree colors names + shows a status letter (M gold, A/U green, D/C red) and rolls child status up to collapsed folders (a dot). Refreshed on mount, manual refresh, window focus, and after file saves (`fsRefreshSignal` bump); a non-git workspace returns `isRepo:false` → clean tree, no errors.
- **File management (VS Code-style explorer).** Create/rename/delete/cut-copy-paste (move/copy) for files + folders. **Backend**: `fstree.js` exports `createEntry`/`renameEntry`/`deleteEntry`/`moveEntry`/`copyEntry` (folders recursive via `cpSync`/`rmSync`), all path-guarded through `resolveInWorkspace` + a `resolveInside` helper that also refuses the workspace root; single-segment name validation (also rejects `.`-prefixed hidden names, which the explorer never lists); moving/copying a folder into itself/a descendant is refused and a move within the same parent is a no-op; paste auto-suffixes on collision (` copy`, ` copy 2`, …), create/rename reject on collision. One dispatch route `POST /api/fs/op` (`{op, …}`) returns the affected workspace-relative path(s). Delete/rename/move also sync the live notebook doc map server-side (`+server.js` calls `notebook.js`'s `dropDocs`/`rekeyDocs`): a deleted path (or any doc nested under a deleted folder) is dropped from the `docs` Map so a later UI/MCP persist can't `writeFileSync`-resurrect the removed file, and a renamed/moved path is rekeyed to its new absolute id (folder ops rekey nested notebook docs too) so edits keep landing in the live doc; the `activePath` pointer follows (reset to null → default notebook when a dropped doc was active). **Frontend**: `Sidebar.svelte` owns the file-ops state (clipboard, inline rename/new, context menu, delete-confirm modal) and exposes it to the recursive `FileTreeNode.svelte` via the `cellarFileOps` **context** (getters forward `$state` so nodes stay reactive — avoids prop-drilling). Inline rename + new-entry use `TreeEntryInput.svelte`. Right-click any node (or empty files-body / root) → context menu; F2/Delete/⌘·Ctrl+C/X/V shortcuts; a cut entry dims until pasted; Paste only on folders/root. After every op the tree + git decorations refresh, and `onFsChange({type,from,path})` bubbles to `+page.svelte` so open tabs are closed (delete) or remapped (rename/move) — no tab left dangling at a gone path.
- **Tabs — session memory + preview.** Open-tab set is persisted per workspace to `localStorage['cellar-tabs:'+workspace]` (paths + preview flag + active id). **First-ever open** (no key) → empty state (`data-testid=empty-state`, no tabs); returning → tabs restored. Single-click a tree file → transient **preview** tab (one at a time; next single-click reuses its slot); **double-click** (tree or tab) or **editing** promotes it to permanent (VS Code semantics). All tab logic lives in `+page.svelte`.
- **Opening a workspace `.ipynb` (live).** Any `.ipynb` opened from the tree loads as a **live, kernel-attached** notebook via `LiveNotebook` (view/edit/run, edits saved back to that file through clean-on-save) — `ipynb`-kind tabs render `LiveNotebook path={tab.path}`, the canonical one uses tab id `notebook`. Preview/permanent tab semantics are unchanged (single-click preview slot, double-click/edit promotes). The old read-only `NotebookFileView` + `/api/fs/notebook` + `readWorkspaceNotebook` were removed.
